Deploy your app to the Google Play Store

An interactive, end-to-end guide: take a working app from "it runs on my emulator" to "anyone can install it from Google Play." Tick off a live readiness checklist, generate your signing config, plan your version bumps, and pick the right testing track — all on this page. The examples are grounded in this repo's AppReleaseBasics project (com.example.appreleasebasics).

Artifact: Android App Bundle (.aab) Account: Play Console ($25, one-time) You'll need: an upload keystore

One honest note up front. Google Play's exact requirements (target API level, testing rules, store-listing forms) change a few times a year. The shape of the process below is stable, but always confirm specifics against the official Play Console guides and the launch checklist before you publish. This page reflects the process as of 2026.

Eight stages turn your build into a published app. Click any stage to jump to it.

3. Stage 1 — Prepare the build

A release build is not just a debug build with a different name. Before you sign anything, set these in your app's build.gradle.kts.

Versioning — the two numbers every release sets

  • versionCode — an integer Google Play uses to decide which build is newer. It must increase on every upload. Users never see it.
  • versionName — a string users see (e.g. "1.0", "1.2.3"). Not used for ordering; bump it however you like (semver is common).

The release build type

Turn on code shrinking/obfuscation (R8) and resource shrinking so your app is smaller and harder to reverse-engineer. In buildTypes.release:

buildTypes {
    release {
        isMinifyEnabled = true          // R8: shrink + obfuscate the code
        isShrinkResources = true        // strip unused resources
        proguardFiles(
            getDefaultProguardFile("proguard-android-optimize.txt"),
            "proguard-rules.pro"
        )
        // signingConfig = signingConfigs.getByName("release")   // wired up in Stage 2
    }
}
Also before you ship: remove noisy debug logging, make sure applicationId is the permanent identity you want (you can never change it after publishing), confirm your targetSdk meets Play's current minimum (Play requires new apps and updates to target a recent API level — within ~a year of the latest Android release), and test a release build on a real device.

4. Stage 2 — Sign your app

Every APK/AAB must be cryptographically signed. Debug builds use a throwaway debug key; a release must be signed with your own key so Google can verify every future update really comes from you.

Upload key vs app signing key (Play App Signing). Today Google recommends Play App Signing: Google holds the real app signing key and signs what users download; you sign your uploads with an upload key you control. The big win: if you ever lose your upload key, you can ask Google to reset it — whereas losing the app signing key (in the old model) meant you could never update the app again. Generate the keystore below and keep it (plus its passwords) somewhere safe and backed up.

5. Stage 3 — Build the release artifact

  
New apps on Google Play must be published as an Android App Bundle (.aab), not an APK. Google
     generates optimized, per-device APKs from your bundle.


  

    
What do I build?

      

        
        
      

    

    

      

        
Android App Bundle — the required format for publishing new apps to Play. You upload one
          .aab; Google builds and signs the smaller APKs each device actually needs.


# Command line — produces app/build/outputs/bundle/release/app-release.aab
./gradlew bundleRelease

        
Android Studio: Build ▸ Generate Signed Bundle / APK ▸
          Android App Bundle, pick your keystore, choose the release build variant.

      

      

        
APK — still fine for sideloading, testing on a device, or distributing outside Play. You
          cannot publish a brand-new app to Google Play as an APK.


# Command line — produces a single installable APK
./gradlew assembleRelease

        
Android Studio: Build ▸ Generate Signed Bundle / APK ▸
          APK. Useful to hand someone a build without Play.

      

    

  


  
Sanity check the bundle before uploading: install it on a device with Google's
    bundletool (bundletool build-apks --mode=universal then install the universal APK), or
    just upload it to the Play Console's internal testing track (Stage 6) and install from there — the fastest
    real-device check.

Required declarations (the forms that trip people up)

  
    
    
  • Content rating — fill the questionnaire; you get an official rating (IARC).
    • 
    
  • Data safety — declare what data your app collects/shares and why. Must match what the app actually does.
    • 
    
  • Privacy policy — a URL is required for most apps (and always if you collect personal/sensitive data).
    • 
    
  • Target audience & content — age groups; extra rules apply if children are in scope.
    • 
    
  • Ads — declare whether the app contains ads.
    • 
    
  • App access — if parts need a login, provide test credentials so reviewers can get in.
    • 
    
  • Pricing & distribution — free/paid and the countries you'll release in.
    • 
  

  
Be truthful on Data safety and permissions. Mismatches between your declarations,
    your privacy policy, and what the app actually does are a leading cause of rejections and takedowns.


  
  
8. Stage 6 — Testing tracks

  
You don't go straight to "everyone." Play gives you four release tracks; promote a build up the
     ladder as your confidence grows.

  

    
1
Internal testing
Up to 100 testers by email; available in minutes. The fastest way to install your signed build on a real device.

    
2
Closed testing
A larger invited group (email lists or Google Groups). Where you run a real beta — and where new personal accounts earn production access.

    
3
Open testing
Anyone with the link can join the beta from your store listing. Good for broad feedback before launch.

    
4
Production
Live on Google Play for everyone in your chosen countries. Supports staged % rollouts.

  


  
New personal developer accounts: accounts created from late 2023 onward generally
    must run a closed test with at least 12 testers, opted in for at least 14 days, before they can apply for
    production access. Plan for this — start your closed test early. (Organization accounts and older accounts may
    differ; check your console.)

9. Stage 7 — Submit, review & roll out

  
Create a release on your chosen track, attach the bundle, and send it for review.

  
    
    
  1. Open the track (e.g. Production or Closed testing) ▸ Create new release.
    2. 
    
  2. Upload your app-release.aab. (First time, accept enrolment in Play App Signing.)
    3. 
    
  3. Set the release name (often the versionName) and write release notes.
    4. 
    
  4. Review release ▸ fix any errors/warnings ▸ Start rollout.
    5. 
    
  5. Google reviews the app. First reviews can take anywhere from a few hours to several days.
    6. 
    
  6. In production, use a staged rollout (e.g. 10% → 50% → 100%) so you can halt if something's wrong.
    7. 
  

  
It's normal to fail a check the first time — Play is strict about the Data safety
    form, target API level, and policy declarations. Read the rejection reason, fix it, and re-submit; you don't start
    over.


  
  
10. Stage 8 — Ship updates

  
Every subsequent release is the same short loop:

  
    
    
  1. Bump versionCode (must increase) and usually versionName — use the
      version helper above.
    2. 
    
  2. Rebuild the signed bundle: ./gradlew bundleRelease.
    3. 
    
  3. Create a new release on a track, upload, add release notes, roll out.
    4. 
  

  
The cardinal rule of updates: sign every update with the same upload key and
    always give it a higher versionCode than anything you've uploaded before. Those two facts are
    what let Google recognize the update as genuinely yours and newer.


  
  
11. Common pitfalls & FAQ

  
Click to expand.

  

    

      
If you enrolled in Play App Signing (the default for new apps), yes: request an
        upload key reset in the Play Console and start signing with a new upload key. If you did not use
        Play App Signing and lost the original app signing key, you cannot update that app — you'd have to publish a new
        listing with a new applicationId. This is exactly why Play App Signing exists.

    

      
Every uploaded build needs a strictly higher versionCode. Bump it in
        build.gradle.kts and rebuild. You can't re-use or lower it, even for a failed upload.

    

      
New apps can't be published as a raw APK. Build with ./gradlew bundleRelease and
        upload the .aab instead of the .apk.

    

      
Play requires new apps and updates to target a recent API level (within about a year of the
        latest Android release). Raise targetSdk (and compileSdk) in
        build.gradle.kts, then re-test for behavior changes the new target opts you into.

    

      
They must (1) be added to the track's tester list, (2) accept the opt-in link the
        console gives you, and (3) install via that link or the Play Store while opted in. Internal testing is the
        quickest track to get a build onto a device.

    

      
First-time reviews and apps touching sensitive areas take longer. Make sure App access (test
        login), Data safety, and content rating are complete and accurate — incomplete declarations stall reviews.
        Don't upload duplicate releases while waiting.

    

      
By default the release build type is not debuggable, which is what you
        want. Don't set isDebuggable = true on release, and don't ship test/debug-only code or logging.
