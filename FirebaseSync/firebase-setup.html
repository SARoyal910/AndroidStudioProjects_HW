Connect FirebaseSync to a real cloud database

Firebase setup & API guide

FirebaseSync is an offline-first notes app. The on-device Room database is the single source of truth the UI always reads; Cloud Firestore is the remote it syncs to. This page walks you, end to end, through standing up Firestore for this exact project — creating the backend, wiring the credentials, the Firebase initialisation code (read line by line), and every Firestore API the app actually calls — plus a tour of the rest of the Firestore toolbox so you know what else is in the box.

Cloud Firestore Firebase BoM 34.15.0 Offline-first Room + sync ~10 minutes
What you build

A Firebase project with one Android app registered to package com.example.firebasesync, a Cloud Firestore database with a notes collection, and the app flipped from its offline Fake cloud to the real Firestore.

How the app uses it

Every note is one Firestore document in the notes collection, keyed by the note's own client-generated id. push = set() that document; pull = a range query on updatedAt.

Zero-setup default

Out of the box the app ships wired to a demo project and also has an in-memory Fake cloud, so it builds, runs, and unit-tests with no network at all. This guide shows how to point it at your own Firestore.

The mental model Room is local and instant; Firestore is the shared remote. The sync engine moves rows between them. Firestore also keeps its own on-device cache, but this project deliberately ignores that and treats Room as the source of truth, so the offline-first mechanics stay explicit. You are configuring the remote, not replacing Room.

Step 1

Create a Firebase project + register the Android app

A Firebase project is the top-level container; inside it you register one Android app for each app that talks to it. We register an Android app whose package name is com.example.firebasesync. Pick whichever route you like — the two produce the same thing.

Route A — the Firebase console (point & click)

  1. Go to console.firebase.google.com and click Add project. Give it any name (e.g. FirebaseSync); Firebase derives a globally-unique project id from it. You can disable Google Analytics — this app does not use it.
  2. On the project dashboard click the Android icon (Add app ▸ Android).
  3. For Android package name enter com.example.firebasesync (this MUST match the app's applicationId). Nickname and SHA-1 are optional for Firestore in this demo — leave them blank.
  4. Firebase offers a google-services.json download. You do not need it for this project — we initialise Firebase in code (see Step 6/7). It is, however, the easiest place to read the three values you do need (Step 5).
  5. Skip the "add the SDK" Gradle steps in the wizard — the dependencies are already in this project (Step 6).

Route B — the Firebase CLI (terminal)

If you have firebase-tools installed, the same setup is a handful of commands. The last one prints the three values you will paste into the app.

terminal — create project & register the Android app 
# Sign in (opens a browser to authorise the CLI)
firebase login

# Create a NEW Firebase project. Pick a globally-unique id; the display name is optional.
firebase projects:create my-firebasesync-id --display-name "FirebaseSync"

# Register an Android app inside that project for our exact package name.
# (Run inside the project, or pass --project my-firebasesync-id.)
firebase apps:create android "FirebaseSync" \
  --package-name com.example.firebasesync \
  --project my-firebasesync-id

# Print the SDK config for the Android app. This is where the THREE values come from:
#   project_id, the Android App ID (mobilesdk_app_id), and the Web API key (current_key).
firebase apps:sdkconfig android --project my-firebasesync-id
Package name must match If the registered package name does not equal the app's applicationId (com.example.firebasesync), Firebase rejects the App ID at runtime. Double-check it now — it is the single most common setup mistake.

Step 2

Enable Cloud Firestore + set security rules

A registered app cannot read or write until the project has a Firestore database and security rules that permit it. Rules are a server-side gatekeeper evaluated on every request — they default to denying everything, so you must open up the notes collection.

Route A — the console

  1. In the left nav open Build ▸ Firestore Database and click Create database.
  2. Choose a location (a multi-region such as nam5 is fine). This is permanent for the database, so pick one near your users.
  3. Start in test mode for the demo (open read/write for ~30 days), or choose production mode and paste the rules below. Either way, the goal is: the notes collection is readable and writable.

The demo rules

Open the Rules tab and use these. They allow anyone to read and write any document in notes — perfect for a learning demo, never for a real app with private data.

firestore.rules — DEMO ONLY 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /notes/{noteId} { allow read, write: if true; }   // DEMO ONLY — tighten before real use
  }
}

Read it top-down: rules version 2; for the Firestore service; for any document under this database; for any document in the notes collection ({noteId} is a wildcard binding the document id), allow both reads and writes unconditionally (if true). In a real app you would replace if true with something like if request.auth != null && request.auth.uid == resource.data.ownerId.

Route B — the CLI

terminal — create the database & deploy rules 
# Create the single default Firestore database in a chosen location.
firebase firestore:databases:create "(default)" --location nam5 \
  --project my-firebasesync-id

# Put the rules above in a file named firestore.rules, point firebase.json at it, then:
firebase deploy --only firestore:rules --project my-firebasesync-id
Do not ship open rules allow read, write: if true means anyone on the internet who knows your project id can read and overwrite every note. It is fine for this teaching demo on a throwaway project. Before any real use, lock it down with Firebase Authentication and per-user ownership checks.

Step 3

The three values you need (and where each comes from)

Because this project initialises Firebase in code (Step 7), you do not drop in a google-services.json file. Instead you copy exactly three values into the constants at the top of data/CloudApi.kt. Here is what each is and where to find it.

Constant in CloudApi.kt Firebase name Where to find it
FIREBASE_PROJECT_ID project_id The Cloud project id (e.g. your-project-id). Project settings ▸ General, or the project_info.project_id field of google-services.json. Routes API calls to the right Firestore.
FIREBASE_APP_ID mobilesdk_app_id The Android App ID, shaped 1:NNN:android:HEX (e.g. 1:000000000000:android:0000000000000000000000). Project settings ▸ Your apps ▸ (the Android app), or client[].client_info.mobilesdk_app_id in google-services.json. Identifies which registered app is calling.
FIREBASE_API_KEY current_key The Web API key (e.g. AIzaSy…). Project settings ▸ Your apps, or client[].api_key[].current_key in google-services.json. Authenticates SDK calls to Google's backend.
Two ways to read them From the console: Project settings (gear icon) ▸ Your apps ▸ select the Android app — the App ID and Web API key are listed there. From the CLI: firebase apps:sdkconfig android prints all three. Either way, download google-services.json if you prefer reading them out of a file.
Are these secrets? The Web API key and App ID are not secrets in the password sense — they ship inside every Android app and are safe to embed. What actually protects your data is your Firestore security rules (Step 2). That is why open demo rules are dangerous and real rules matter.

Step 4

The Gradle dependencies

The Firestore libraries are already declared in this project, so there is nothing to add — but it is worth understanding what each line does. The dependencies live in two files: the version catalog (gradle/libs.versions.toml) and the module build script (build.gradle.kts).

The version catalog

gradle/libs.versions.toml 
[versions]
# Firebase BoM = ONE version number that aligns every Firebase library (here: Cloud Firestore).
# The individual firebase-* deps below are then declared WITHOUT a version — the BoM picks it.
firebaseBom = "34.15.0"
coroutines = "1.10.2"

[libraries]
# --- Cloud transport: Google Cloud Firestore (the REAL cloud database) ---
# firebase-bom is a "platform" dependency; firebase-firestore takes its version from it.
firebase-bom = { group = "com.google.firebase", name = "firebase-bom", version.ref = "firebaseBom" }
firebase-firestore = { group = "com.google.firebase", name = "firebase-firestore" }
# Bridges Firebase's Play-services Task<T> to Kotlin coroutines: lets us write `task.await()`.
kotlinx-coroutines-play-services = { group = "org.jetbrains.kotlinx", name = "kotlinx-coroutines-play-services", version.ref = "coroutines" }

The module build script

build.gradle.kts 
dependencies {
    // Cloud transport: Google Cloud Firestore is the REAL cloud database.
    implementation(platform(libs.firebase.bom))           // one aligned set of Firebase versions
    implementation(libs.firebase.firestore)               // the Firestore client (Kotlin extensions bundled in)
    implementation(libs.kotlinx.coroutines.play.services) // lets us `await()` a Firestore Task in a suspend fun
}

Why the BoM?

A BoM (Bill of Materials) is a "platform" dependency: you pin one version (34.15.0) and every individual firebase-* library you add is declared without a version — the BoM chooses a set that is guaranteed to work together. That is why firebase-firestore has no version of its own. Add more Firebase products later (Auth, Storage, Functions) and they all inherit aligned versions from the same BoM.

The design choice: initialise Firebase in code, no plugin file

Notice what is not here: there is no com.google.gms.google-services Gradle plugin and no google-services.json in the module. This project hands the three credentials directly to FirebaseOptions.Builder() in Kotlin (Step 7), so it compiles and runs with zero extra files. That makes it trivial to clone, build, and unit-test, and it means a student goes live with a single one-line edit.

The production-standard alternative is equivalent The usual approach is to drop your google-services.json into the module and apply the com.google.gms.google-services plugin, which reads that file at build time and wires the same three values into FirebaseOptions automatically. Both routes produce the identical configuration — they just differ in where the values come from (a JSON file vs three Kotlin constants).

Step 5

The initialisation code, line by line

This is the heart of the connection: the few lines that turn three constants into a live Firestore client. It lives at the top of FirestoreCloudApi in data/CloudApi.kt and runs once, lazily, the first time the real cloud is used.

The three credential constants

data/CloudApi.kt — paste YOUR values here 
private const val FIREBASE_PROJECT_ID = "your-project-id"
private const val FIREBASE_APP_ID = "1:000000000000:android:0000000000000000000000"
private const val FIREBASE_API_KEY = "AIzaSy-REPLACE-WITH-YOUR-WEB-API-KEY"

These are the demo project's values. Replace them with your project_id, Android App ID, and Web API key from Step 5 to point the app at your own Firestore.

Building the Firestore client

data/CloudApi.kt — class FirestoreCloudApi(context: Context) 
private val db: FirebaseFirestore = run {
    val app: FirebaseApp = FirebaseApp.getApps(context).firstOrNull()
        ?: FirebaseApp.initializeApp(
            context,
            FirebaseOptions.Builder()
                .setProjectId(FIREBASE_PROJECT_ID)   // which Cloud project to use
                .setApplicationId(FIREBASE_APP_ID)   // which Android app within that project
                .setApiKey(FIREBASE_API_KEY)         // API key for SDK authentication
                .build(),                            // produces the immutable FirebaseOptions
        )
    FirebaseFirestore.getInstance(app)               // the Firestore client handle for this app
}
API📞 Firebase API · FirebaseApp.initializeApp() API📞 Firebase API · FirebaseFirestore.getInstance() Firebase🔌 Connecting to Firebase · no data written yet (no socket opens until the first read/write)

Read it as four moves:

  1. FirebaseOptions.Builder() … .build() — the programmatic equivalent of google-services.json. Three setters carry the three values: setProjectId() tells the SDK which Cloud project to route to, setApplicationId() says which registered Android app is calling, and setApiKey() authenticates SDK calls to Google's backend. .build() freezes them into an immutable FirebaseOptions value object.
  2. FirebaseApp.getApps(context).firstOrNull() — a FirebaseApp is a process-wide singleton. getApps() returns every app already registered in this process; we take the first if one exists. This guards against initialising twice.
  3. ?: FirebaseApp.initializeApp(context, options) — only if no app exists yet do we register one. Calling initializeApp() a second time with the same default name would throw, which is exactly why the getApps().firstOrNull() ?: guard is there: initialise once, reuse forever.
  4. FirebaseFirestore.getInstance(app) — returns the Firestore client bound to that app. It is lazy: no network socket opens here. The connection is established only on the first real read or write.
Why run { … }? run { … } is a Kotlin scope function that runs a block and returns its last expression. It lets us declare a few local helpers (app) and still assign a single value (the Firestore client) to db in one initialiser. Because db is a property of the class, the whole block runs the first time FirestoreCloudApi is constructed.
A benign Logcat warning On a fresh install you may see "FirebaseApp initialization unsuccessful" in Logcat. That comes from Firebase's own auto-init ContentProvider, which fires before our code runs and finds no google-services.json. It is harmless: we initialise Firebase ourselves from the constants above, and real-mode sync works fine. (In Fake mode Firebase is never touched at all.)

Step 6

Flip the Real/Fake switch

The whole app talks to a tiny CloudApi interface with two methods, push and pullSince. Behind it sits either the real FirestoreCloudApi or an in-memory FakeCloudApi. One boolean chooses which. Set useFakeCloud = false to go live.

FirebaseSyncViewModel.kt — the single lever 
// useFakeCloud = false → talk to the REAL Cloud Firestore configured in data/CloudApi.kt.
// Flip back to NoteRepository.get(app) (default true) to demo fully offline against the Fake.
private val repo = NoteRepository.get(app, useFakeCloud = false)

That flag flows straight through to the factory at the bottom of CloudApi.kt, which is where the choice is actually made:

data/CloudApi.kt — the factory 
fun provideCloudApi(context: Context, useFake: Boolean = true): CloudApi =
    if (useFake) fakeCloud else FirestoreCloudApi(context)
ModeWhat it doesUse it for
Fake (default) An in-memory "server" that imitates latency, last-write-wins, and even another device's edits — with no network. Seeds a Welcome note on first sync. The whole sync engine runs offline. Building, demos with no setup, unit tests.
Real FirestoreCloudApi talks to your live Cloud Firestore using the three constants. Requires the INTERNET permission (already declared) and a networked device. Seeing data appear in the Firestore console and sync across devices.
Default Fake keeps the app testable Because the default is useFakeCloud = true, the project compiles, runs, and passes its unit tests with no Firebase project and no connection. Flipping to false is the only change required to go live once your three values are in place.

Action → code → Firestore call

What fires when you add, edit, delete, or sync — and the exact Firestore call

This section follows a single user tap all the way down: from the button in MainActivity.kt, through the ViewModel and the Repository, into Room, and — only during sync — to the precise Cloud Firestore API that fires. The headline to keep in mind: tapping Add / Edit / Delete writes to Room first and shows instantly; Firestore is touched only by the push/pull inside sync(); and drawing the list touches no Firestore at all.

Operation You tap… Code path (ViewModel → Repository) Firestore API called Effect
Add "Add note" addNote() → writeThenSync { repo.addNote() } → dao.upsert(PENDING) → later sync() → cloud.push(dto) notes.document(id).set(note).await() Creates the document at notes/{id}
Edit / Update "Save edit" editNote(id, …) → writeThenSync { repo.editNote() } → dao.upsert(PENDING) → sync() → cloud.push(dto) notes.document(id).set(note).await() (same call) Overwrites the document = update
Delete "Delete" deleteNote(id) → writeThenSync { repo.deleteNote() } → dao.upsert(deleted=true, PENDING) → sync() → cloud.push(dto) → dao.hardDelete(id) notes.document(id).set(note).await() (writes deleted=true) Overwrites with a tombstone (soft delete)
Sync / Pull "Sync now" syncNow() → requestSync() → sync() → cloud.pullSince(maxUpdatedAt) whereGreaterThan + orderBy + get + toObject Reads remote changes into Room
Read / Display nothing — automatic repo.notes ← dao.observeVisibleNotes() → StateFlow → Compose none — reads Room Draws the list locally, offline, instantly
How to read each trace below Each card starts with the user action, then walks the code hops (file · function with a real snippet), and ends with a highlighted "Firestore API called" line stating exactly which API fires and whether it creates, overwrites (updates), or reads. Firebase🗑️ Deleting from Firebase · marks notes/{id} deleted (tombstone — soft delete) API📞 Firebase API · DocumentReference.set(note) (deleted=true)
The unused alternative: a hard remote delete Firestore does have a real remove — notes.document(id).delete().await() — which vanishes the document server-side. This app deliberately does not call it. Why a tombstone instead? Because a hard delete leaves nothing to push: an offline delete would have no row to upload, and other devices would never hear about it. A deleted = true tombstone is a normal pushable row, so an offline delete still syncs and propagates. See section 11 for the delete() snippet.

    MainActivity.kt
    OutlinedButton(onClick = { vm.syncNow() }) { Text("Sync now") }
  2. 2
    FirebaseSyncViewModel.kt · syncNow() → requestSync()
    syncNow() just calls requestSync(), which enqueues the durable worker and, when online, runs repo.sync() on the spot.
    FirebaseSyncViewModel.kt
    fun syncNow() = requestSync()

private fun requestSync() {
    SyncScheduler.requestSync(getApplication())     // durable background retry path
    if (connectivity.currentlyOnline()) {
        viewModelScope.launch { runCatching { repo.sync() } }
    }
}
  3. 3
    NoteRepository.kt · sync() — the PULL step
    After the push loop, sync computes the cursor (maxUpdatedAt()) and asks the cloud for everything newer, merging each via last-write-wins.
    NoteRepository.kt · sync()
    val since = dao.maxUpdatedAt() ?: 0L
for (remote in cloud.pullSince(since)) {
    val local = dao.findById(remote.id)
    if (shouldAcceptRemote(local, remote)) {        // last-write-wins
        if (remote.deleted) dao.hardDelete(remote.id)
        else dao.upsert(remote.toEntity(SyncState.SYNCED))
    }

The real code

The Firestore APIs this app uses

The whole Firestore surface this app touches fits in two methods of FirestoreCloudApi. The frame to keep in your head:

The two operations push() = set() one document.  ·  pull() = whereGreaterThan + orderBy + get + toObject.

Below is one block per API: the real snippet from CloudApi.kt, a plain explanation, and which method it lives in.

db.collection("notes") → CollectionReference push & pull (shared)
data/CloudApi.kt
private val notes get() = db.collection("notes")

Returns a CollectionReference — a lightweight path handle to the top-level notes collection. No network call happens here; it is just a descriptor that the document and query calls hang off of. It is a get property, so each access yields a fresh (cheap) reference.

Called when you… add, edit, delete, or sync — every push() and pullSince() starts from this reference (it is the root both operations hang off). See the traces in section 9.

API📞 Firebase API · FirebaseFirestore.collection("notes") Firebase∅ Not sent to Firebase · local only (path handle, no request)
notes.document(note.id) → DocumentReference push()
data/CloudApi.kt — push()
notes.document(note.id)  // DocumentReference at notes/{id}

Returns a DocumentReference at the path notes/{id} — again just a path handle, no network. Using the note's own client-generated id as the document id means we never ask Firestore to auto-generate one: the same UUID the phone created offline is the key everywhere.

Called when you… add, edit, or delete — it is the first half of every push(), naming the exact document that set() is about to write.

API📞 Firebase API · CollectionReference.document(id) Firebase∅ Not sent to Firebase · local only (path handle, no request)
.set(note) → Task<Void> push()
data/CloudApi.kt — push()
notes.document(note.id)  // DocumentReference at notes/{id}
    .set(note)           // create-or-overwrite the document; returns Task<Void>
    .await()             // suspend until the server confirms; throws on network/permission error
return note              // return our copy as the "server's authoritative copy"

set() creates the document if it does not exist or completely overwrites it if it does. It does not compare timestamps or versions — it always wins. It returns a Task<Void> that resolves when the server confirms the write (or rejects on a network / permission error). This is the entire push.

Called when you… add (creates), edit (overwrites = update), or delete (overwrites with a deleted=true tombstone). All three user actions push through this one set() — see traces 1, 2, and 3 in section 9.

API📞 Firebase API · DocumentReference.set(note) Firebase➕ Adding to Firebase · writes the notes/{id} document (new id) Firebase✏️ Updating Firebase · overwrites the notes/{id} document Firebase🗑️ Deleting from Firebase · marks notes/{id} deleted (tombstone — soft delete)
.whereGreaterThan("updatedAt", since) → Query pullSince()
data/CloudApi.kt — pullSince()
.whereGreaterThan("updatedAt", since)          // range filter: only newer documents

Adds a range filter: return only documents whose updatedAt is strictly greater than since (our local pull cursor — the epoch-millis of the newest note already in Room). Field names are strings and must match NoteDto's property names exactly. The result is a Query (still no network yet).

Called when you… sync/pull — it builds the read query in pullSince() (trace 4 in section 9). Never on add/edit/delete.

API📞 Firebase API · CollectionReference.whereGreaterThan("updatedAt", since) Firebase∅ Not sent to Firebase · local only (builds a Query, no request yet)
.orderBy("updatedAt", Query.Direction.ASCENDING) → Query pullSince()
data/CloudApi.kt — pullSince()
.orderBy("updatedAt", Query.Direction.ASCENDING) // oldest-first; same field → no composite index needed

Sorts results oldest-first so we merge them in chronological order. Firestore requires an orderBy() on the same field used in a range filter. Because the range filter and the orderBy target the same field (updatedAt), Firestore satisfies it with an auto-created single-field index — no composite index is needed. (Range + order on different fields would require you to create a composite index.)

Called when you… sync/pull — chained onto the same query in pullSince() (trace 4 in section 9).

API📞 Firebase API · Query.orderBy("updatedAt", ASCENDING) Firebase∅ Not sent to Firebase · local only (still building the Query)
.get() → Task<QuerySnapshot> pullSince()
data/CloudApi.kt — pullSince()
val snapshot = notes
    .whereGreaterThan("updatedAt", since)
    .orderBy("updatedAt", Query.Direction.ASCENDING)
    .get()                                         // returns Task<QuerySnapshot> (async, not a coroutine)
    .await()

get() finally executes the query against the Firestore backend and returns a Task<QuerySnapshot>. A Task is the Play-services async type — a callback-based future, not a Kotlin coroutine. This is the one-shot read; for live updates you would use a listener instead (see the next section).

Called when you… sync/pull — this is the moment the read actually hits the network, inside pullSince() (trace 4 in section 9).

API📞 Firebase API · Query.get() Firebase⬇️ Reading from Firebase · fetches the notes documents (matching the query)
kotlinx.coroutines.tasks.await() push() & pullSince()
data/CloudApi.kt — import
import kotlinx.coroutines.tasks.await   // turns a Firebase Task<T> into a suspend-friendly await()

From the kotlinx-coroutines-play-services bridge library (Step 6). .await() suspends the current coroutine until the Task completes — or throws if the network fails or Firestore returns an error. It is what lets us write straight-line sequential code inside a suspend fun instead of nesting callbacks. The exception path is how a failed push/pull bubbles up to the repository and triggers a WorkManager retry.

Called when you… add, edit, delete, or sync — it waits on both the set() in push() and the get() in pullSince(), so every operation except Read passes through an await().

API📞 Firebase API · Task.await() Firebase∅ Not sent to Firebase · local only (suspends; the wrapped call already did the work)
snapshot.documents.mapNotNull { it.toObject(NoteDto::class.java) } pullSince()
data/CloudApi.kt — pullSince()
return snapshot.documents.mapNotNull { it.toObject(NoteDto::class.java) }

snapshot.documents is a List<DocumentSnapshot>, one per matched document. it.toObject(NoteDto::class.java) is Firestore's reflective deserialiser: it calls NoteDto's no-arg constructor and then sets each field by matching document field names to property names. mapNotNull drops any document that fails to map, yielding a plain List<NoteDto> the repository merges into Room.

Called when you… sync/pull — it deserialises the documents the get() returned, the last step of pullSince() (trace 4 in section 9).

API📞 Firebase API · DocumentSnapshot.toObject(NoteDto::class.java) Firebase∅ Not sent to Firebase · local only (maps the already-fetched documents)
Why NoteDto needs all-default properties Firestore's toObject() needs a no-argument constructor, and a Kotlin data class only generates one when every parameter has a default. That is exactly why NoteDto is written this way:
data/Note.kt
data class NoteDto(
    val id: String = "",
    val title: String = "",
    val body: String = "",
    val updatedAt: Long = 0L,
    val deleted: Boolean = false,
)
Drop the defaults and toObject() throws at runtime. (No annotation is needed — Firestore does its own object mapping by reflection.)

Good to know

The rest of the Firestore toolbox

None of these are used in this app The snippets below are illustrative — they are not in FirebaseSync. They are a quick tour of what else Firestore offers and how each would change this app if you reached for it.

Real-time listeners (vs our polling pull)

illustrative — NOT in this app
// Push-based: Firestore calls you back whenever any matching document changes.
val registration = notes.addSnapshotListener { snap, e ->
    if (e != null) return@addSnapshotListener
    val dtos = snap?.documents?.mapNotNull { it.toObject(NoteDto::class.java) }.orEmpty()
    // …upsert dtos into Room as they arrive…
}
// registration.remove()  // stop listening

A listener turns sync from pull-when-asked into pushed-as-it-happens. It would replace our manual pullSince().get() with a stream that fires on every remote change — at the cost of an always-open connection and lifecycle management (you must remove() it).

Delete a document

illustrative — NOT in this app
notes.document(id).delete().await()   // hard-delete the document on the server

This app never hard-deletes remotely — it pushes a deleted = true tombstone via set() so the delete replicates to other devices. A real delete() would vanish the row server-side with no tombstone to propagate.

Update specific fields

illustrative — NOT in this app
notes.document(id).update(mapOf("title" to "Renamed")).await()  // patch only these fields

update() patches the named fields and leaves the rest untouched, unlike set() which overwrites the whole document. The app uses whole-document set() because a note is small and last-write-wins is simpler that way.

Batched writes

illustrative — NOT in this app
val batch = db.batch()
batch.set(notes.document(a.id), a)
batch.set(notes.document(b.id), b)
batch.commit().await()   // all-or-nothing: every write applies, or none does

A WriteBatch commits many writes atomically in one round trip. Our app pushes one pending row at a time; a batch would let it flush the whole outbox in a single atomic call.

Transactions

illustrative — NOT in this app
db.runTransaction { tx ->
    val current = tx.get(notes.document(id))           // read inside the transaction
    val n = (current.getLong("version") ?: 0L) + 1
    tx.update(notes.document(id), mapOf("version" to n)) // conditional write
}.await()

A transaction reads-then-writes atomically and retries on contention — the server-side way to resolve conflicts. The app instead uses a client-side last-write-wins rule (shouldAcceptRemote()), which is enough for a single-user demo.

Pagination

illustrative — NOT in this app
val first = notes.orderBy("updatedAt").limit(20).get().await()
val last = first.documents.last()
val nextPage = notes.orderBy("updatedAt").startAfter(last).limit(20).get().await()

limit(n) caps the page size; startAfter(doc) resumes after the last document of the previous page (cursor pagination). Our pull is bounded by the updatedAt cursor instead, so a single pull is naturally small.

Server timestamps

illustrative — NOT in this app
notes.document(id).update(mapOf("updatedAt" to FieldValue.serverTimestamp())).await()

FieldValue.serverTimestamp() stamps the field with the server's clock, not the device's. The app uses the device clock for updatedAt so it can stamp rows while completely offline; a server timestamp would give one authoritative clock but only after the write reaches the server.

Firestore's own offline persistence

illustrative — NOT in this app
// Firestore can cache locally and queue writes by itself (enabled by default on Android).
val settings = firestoreSettings {
    setLocalCacheSettings(persistentCacheSettings { })   // on-disk cache + offline write queue
}
db.firestoreSettings = settings
// snapshot.metadata.isFromCache  // tells you a read came from the local cache, not the server

Firestore has its own offline cache and write queue. This project deliberately ignores it and makes Room the single source of truth, so the offline-first mechanics (outbox, tombstones, last-write-wins) stay explicit and identical to CloudSync. If you leaned on Firestore's cache instead, much of the sync engine would disappear — at the cost of giving up that explicit control.

Write and run tests

Testing

This project keeps its logic small and pure so most of the interesting behaviour can be verified with plain JVM unit tests — no device, no emulator, no Firebase account needed.

What's already tested — and why it needs no Firebase

Open src/test/java/com/example/firebasesync/FirebaseSyncUnitTest.kt. Every test in it runs on the plain JVM (the host machine). Firebase is never touched because the app's sync logic is tested through two seams:

  • shouldAcceptRemote(...) in NoteRepository.kt — a pure Kotlin function with no dependencies. Five test cases cover all four branches of the last-write-wins rule: no local copy (always accept), local SYNCED + remote newer (accept), local SYNCED + same timestamp (accept — idempotent re-pull), local SYNCED but newer (reject), and local PENDING (reject — the unpushed edit wins on the client side).
  • Converters round-trip — Room's @TypeConverter serialises the SyncState enum to a String and back. The test checks every enum entry round-trips without data loss and that an unknown string falls back to the safe default (SYNCED).
  • FakeCloudApi push → pull round-trip — the in-memory "server" that imitates Firestore. Tests push a note, then pull it back; push a stale version and confirm last-write-wins keeps the newer one; and use a future cursor to confirm only newer rows are returned.
∅ Not sent to Firebase — these run fully offline FakeCloudApi lives in memory. No credentials, no network request, no device. shouldAcceptRemote is a pure function: it just takes two Kotlin values and returns a Boolean. These tests run on CI, on a plane, and inside testDebugUnitTest in under a second each.

Write a new unit test

New tests go in the same file (or a new .kt file next to it in src/test/). The pattern used throughout FirebaseSyncUnitTest.kt is:

Pure, no-coroutine test — for shouldAcceptRemote and anything else that is a plain function:

FirebaseSyncUnitTest.kt — pure @Test (no coroutines) 
// @Test marks the method as a test case JUnit 4 should discover and run.
// No parameters, no return value — just assertions.
@Test fun rejectRemote_whenLocalHasUnpushedEdit() {
    // shouldAcceptRemote is a pure top-level function in NoteRepository.kt.
    // No Room, no network — just two Kotlin values in, one Boolean out.
    assertFalse(shouldAcceptRemote(
        local  = Note(id = "a", title = "t", body = "b",
                      updatedAt = 100, syncState = SyncState.PENDING),
        remote = NoteDto(id = "a", title = "t", body = "b", updatedAt = 999)
    ))
    // assertFalse(value) fails the test if value is true.
    // assertEquals(expected, actual) checks two values are equal — same family.
}

Coroutine test — for FakeCloudApi and any suspend function. runTest { } from kotlinx-coroutines-test provides a test coroutine scope that runs the whole body synchronously (it advances virtual time so delay() inside the fake doesn't make the test slow):

FirebaseSyncUnitTest.kt — coroutine test with runTest { } 
// runTest { } is provided by kotlinx-coroutines-test (already in testImplementation).
// It gives you a coroutine scope so you can call suspend functions in a test.
// Virtual time is used, so delay() inside FakeCloudApi returns immediately.
@Test fun fakeCloud_pushThenPull_returnsTheRow() = runTest {
    val cloud = FakeCloudApi()                        // create a fresh in-memory "server"
    cloud.push(NoteDto(id = "x", title = "hi",        // push one note
                       body = "there", updatedAt = 1_000))
    val pulled = cloud.pullSince(0)                   // pull everything since epoch 0
    // assertTrue(message, condition) — fails with the message if condition is false.
    assertTrue("pushed row should come back on pull", pulled.any { it.id == "x" })
}

Integration-testing against real Firestore — use the Emulator, not prod

When you want to test code that calls FirestoreCloudApi directly (i.e., actual Firestore SDK calls, not FakeCloudApi), do not run against your live project. Use the Firebase Local Emulator Suite instead: it is free, runs locally, resets between runs, and lets you seed data or inspect writes without touching production.

Step 1 — install & start the Firestore emulator

You need firebase-tools installed (npm i -g firebase-tools).

terminal — install and start the Firestore emulator 
# One-time project setup — creates firebase.json + .firebaserc in the project root.
firebase init emulators

# Start only the Firestore emulator (listens on port 8080 by default).
firebase emulators:start --only firestore

# The Emulator UI is served at http://localhost:4000 — you can browse documents there.
Step 2 — point the app at the emulator in test/debug builds

The Android emulator's loopback address for the host machine is 10.0.2.2 (not localhost, because the Android emulator is its own VM). Port 8080 is the Firestore emulator's default.

data/CloudApi.kt — redirect Firestore to the emulator (test/debug only) 
// Call this ONCE after FirebaseFirestore.getInstance(app) to redirect all SDK
// calls to the local emulator instead of the real Firestore backend.
// 10.0.2.2 = the host machine's loopback address as seen from inside the Android emulator.
// 8080     = the default Firestore emulator port (set by firebase init emulators).
// IMPORTANT: only call this in test or debug builds — never in a release build.
val db: FirebaseFirestore = FirebaseFirestore.getInstance(app)
db.useEmulator("10.0.2.2", 8080)

With the emulator running and the client pointing at it you can:

  • Run the app normally against a fresh, isolated Firestore — all reads and writes go to the local process, not the cloud.
  • Seed data by writing documents directly in the Emulator UI at http://localhost:4000, or by calling cloud.push(...) in a test setup block.
  • Run ./gradlew :connectedDebugAndroidTest (with the emulator running) for fully integrated DAO + Firestore SDK tests.
  • Each firebase emulators:start run begins with a blank database, so tests never interfere with each other or with production data.
This is separate from FakeCloudApi FakeCloudApi replaces the Firestore SDK entirely (pure Kotlin, runs on the JVM, no emulator needed). The Firebase Local Emulator Suite lets you test the real FirestoreCloudApi code — the same set(), whereGreaterThan(), and .await() calls that ship in production — against a safe, local Firestore process. Use the Fake for fast, offline unit tests; use the emulator when you specifically need to exercise the Firestore SDK itself.

When it doesn't work

Troubleshooting

SymptomCause & fix
Logcat: "FirebaseApp initialization unsuccessful" Benign. Firebase's auto-init ContentProvider runs before our code and finds no google-services.json. We initialise Firebase manually from the constants; ignore this warning. (It never appears in Fake mode.)
PERMISSION_DENIED on read or write Your security rules deny the operation. For the demo, set the notes rule to allow read, write: if true (Step 2). For real apps, make sure the request satisfies your auth/ownership conditions.
UNAVAILABLE / "Unable to resolve host" A network problem or a wrong project id. Confirm the device is online, that FIREBASE_PROJECT_ID matches your project, and that a (default) Firestore database actually exists.
toObject() returns objects full of nulls / zeros Either the Firestore field names don't match NoteDto's property names, or NoteDto lost its default values. Every property must have a default so Kotlin generates the no-arg constructor Firestore needs.
Crash: FirebaseApp name [DEFAULT] already exists You called initializeApp() twice. The FirebaseApp.getApps(context).firstOrNull() ?: guard in CloudApi.kt exists precisely to call initializeApp() once — make sure you didn't also add the google-services plugin (which auto-inits too) on top of the in-code init.
Query needs an index Range filter and orderBy on the same field (our case) need no composite index. If you add a range on one field and orderBy on another, Firestore returns an error with a one-click link to create the composite index.
Still stuck? Fall back to Fake Flip useFakeCloud = true to rule out Firebase entirely. If the app works fully against the Fake cloud, the problem is in your Firebase project / rules / three values, not in the app's sync engine.
