Cloud storage, in conjunction with Room

How a note syncs

FirebaseSync shows the pattern every real app eventually needs: a local database (Room) kept in sync with a Google Cloud Firestore database, that keeps working when the device is offline. The trick is one rule — Room is the single source of truth — plus a background SyncWorker that reconciles it with Firestore. This page is that design as blocks of real code, from the entity's sync fields to last-write-wins conflict resolution.

Step 0 — wiring the offline-first stack

Four pieces: Room (+ KSP) for the local cache, Cloud Firestore (+ Firebase BoM + coroutines bridge) for the real cloud, and WorkManager for background sync.

Predict before reveal: why does build wiring matter before any note exists?
Room, Cloud Firestore, the coroutines bridge, and WorkManager are the infrastructure. The app can only save locally, talk to a real Firestore database, suspend on cloud calls, and retry sync if these libraries are present.

Step 1 · the row, plus sync bookkeeping

The entity — a normal note + the offline-first fields

Think of the entity as the shape of one saved note in the local database. Room stores this row on the device. FirebaseSync adds a few extra fields so the app can tell whether that row still needs to be sent to the cloud.

PiecePlain meaningWhy it matters
@EntityThis tells Room: "save this class as a table row."The app can keep notes on the phone, even with no network.
data classA Kotlin class mostly used to hold data.Note holds the fields for one saved note.
@PrimaryKeyThe field Room uses as the row's unique identity.Room can find, replace, or delete the exact note by its id.
idThe note's permanent name tag. FirebaseSync makes it on the phone.A note made offline already has an id, so it can be edited, deleted, and uploaded later.
UUIDA randomly generated id that is extremely unlikely to collide.The phone can safely create note ids before the server sees the note.
updatedAtThe time this note was last changed.If two copies disagree, FirebaseSync can pick the newer one.
enum classA Kotlin type with a fixed list of allowed values.SyncState can only be SYNCED, PENDING, or FAILED.
syncStateThe note's upload status: SYNCED, PENDING, or FAILED.PENDING means "still needs to go to the cloud." The UI badge comes from this.
deletedA hidden "please delete this" marker.If the user deletes offline, the app must remember that delete and send it later.
@SerializableThis tells Kotlin how to turn the class into JSON and back.Not used in FirebaseSync — Firestore maps documents with its own reflection; NoteDto is a plain data class with default values.
NoteDtoThe version of a note that is sent over the network.The database shape and the network shape stay separate, then mappers copy data between them.
data/Note.kt
enum class SyncState { SYNCED, PENDING, FAILED }   // the outbox marker

@Entity(tableName = "notes")
data class Note(
    @PrimaryKey val id: String,            // CLIENT-generated UUID — an offline row has an id before the server sees it
    val title: String,
    val body: String,
    val updatedAt: Long,                   // the clock for last-write-wins conflict resolution
    val syncState: SyncState = SyncState.PENDING,
    val deleted: Boolean = false,          // soft-delete tombstone — a delete you can still PUSH while offline
)

// NoteDto has NO @Serializable — Firestore maps documents with its own reflection, not JSON.
// Every property MUST have a default value so Firestore can call the no-arg constructor.
data class NoteDto(
    val id: String = "",
    val title: String = "",
    val body: String = "",
    val updatedAt: Long = 0L,
    val deleted: Boolean = false,
)

fun Note.toDto() = NoteDto(id, title, body, updatedAt, deleted)
fun NoteDto.toEntity(state: SyncState = SyncState.SYNCED) = Note(id, title, body, updatedAt, state, deleted)
  • Client-generated id — the phone makes the id right away, so the app does not need to wait for the server.
  • updatedAt — the "last changed" time used when two devices edited the same note.
  • syncState — tells the app whether this row is already in the cloud, still waiting, or failed.
  • deleted tombstone — a delete is saved as a hidden row first, then removed after the cloud receives it.
  • Default values on NoteDto — required so Firestore's object mapper can deserialise documents using the no-arg constructor.
Predict before reveal: why should the phone create the note id before upload?
An offline note still needs an identity. The user can edit or delete it before the server ever sees it, and the sync code can upload that exact same row later.

Step 2 · reads for the UI, queries for the engine

The DAO

The DAO is the database doorway. All reads and writes to the Room table go through it. Some DAO functions are for the screen. Others are for sync.

What Flow means here A Flow is a live stream of values. It is not just one database answer. It is more like a subscription: the screen says, "tell me the current list of notes, and tell me again any time that list changes." Room sends a new value through the Flow whenever the matching rows in the database change.

One source of truthThe screen follows Room instead of keeping a separate copy.If sync pulls a cloud note into Room, the Flow emits and the UI redraws from Room.
Works offlineThe stream comes from the local database, not the network.Airplane mode does not stop observeVisibleNotes(); local Room changes still emit.

What just happened

    Recent emissions
      UI-facing queries

      These feed the screen. They return Flows, so Room can keep sending updated values after the first answer.

      • observeVisibleNotes() gets notes the user should see. Deleted tombstones stay hidden.
      • observePendingCount() counts notes still waiting to sync.
      • Both are live: when Room changes, the next value flows to the ViewModel, then Compose redraws the screen.
      Sync-engine queries

      These help the sync code answer two questions: "what do I need to upload?" and "what new cloud rows should I download?"

      • notesToPush() returns the outbox: rows that are not SYNCED yet.
      • maxUpdatedAt() gives the newest local change time, so the app can ask the cloud for newer rows only.
      • suspend means the function may pause while waiting for database or network work, without freezing the screen.
      DAO code wordPlain meaningFirebaseSync example
      @DaoThis marks the interface as a Room database access object.Room generates the real implementation of NoteDao.
      @QueryA SQL database command written above a Kotlin function.observeVisibleNotes() runs a query for non-deleted notes.
      @InsertA Room write command for saving a row.upsert(note) saves a note row into Room.
      OnConflictStrategy.REPLACEIf a row with the same primary key already exists, replace it.Saving a pulled cloud copy updates the existing local row.
      cursorA saved position used to ask for only newer data.maxUpdatedAt() is the pull cursor: "give me cloud notes newer than this."

      data/NoteDao.kt
      @Dao
interface NoteDao {
    // --- reactive reads for the UI ---
    @Query("SELECT * FROM notes WHERE deleted = 0 ORDER BY updatedAt DESC")
    fun observeVisibleNotes(): Flow<List<Note>>            // tombstones hidden

    @Query("SELECT COUNT(*) FROM notes WHERE syncState != 'SYNCED'")
    fun observePendingCount(): Flow<Int>                    // drives "N pending"

    // --- sync-engine queries ---
    // suspend = this may pause while Room reads or writes on disk.
    // Call suspend functions from a coroutine, not directly from Compose.
    @Query("SELECT MAX(updatedAt) FROM notes")
    suspend fun maxUpdatedAt(): Long?                       // the pull cursor

    @Query("SELECT * FROM notes WHERE syncState != 'SYNCED'")
    suspend fun notesToPush(): List<Note>                   // the OUTBOX

    @Query("SELECT * FROM notes WHERE id = :id")
    suspend fun findById(id: String): Note?

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun upsert(note: Note)

    @Query("DELETE FROM notes WHERE id = :id")
    suspend fun hardDelete(id: String)                     // only after the cloud confirms a tombstone
}
      Predict before reveal: which DAO query is the outbox?
      notesToPush() is the outbox query. It returns every row whose syncState is not SYNCED, which means the row still needs cloud work.

      Step 3 · the cloud, Real and Fake

      The cloud doorway — fake or real behind the same functions

      CloudApi is the small doorway to the cloud. The Repository does not need to know if the cloud is Google Cloud Firestore or the offline fake. It only needs two actions: send one note up, and ask for newer notes back.

      MethodPlain EnglishWhy it matters
      APIA small set of functions one part of the app promises other parts can call.CloudApi promises push and pullSince.

      interface CloudApi {
    // suspend = this may pause for network work without blocking the UI thread.
    suspend fun pullSince(since: Long): List<NoteDto>   // remote rows newer than the cursor
    suspend fun push(note: NoteDto): NoteDto            // upsert; returns the server's copy
}

// THE SWITCH — fake (default, offline) uses one shared in-memory server while the app process is running.
fun provideCloudApi(context: Context, useFake: Boolean = true): CloudApi =
    if (useFake) fakeCloud else FirestoreCloudApi(context)
      data/CloudApi.kt · FakeCloudApi (offline server)
      class FakeCloudApi : CloudApi {
    private val server = linkedMapOf<String, NoteDto>()   // the pretend server's store

    init {
        // Seed one row so a brand-new install pulls a "Welcome to FirebaseSync" note on first sync.
        val seededAt = System.currentTimeMillis() - 1_000
        server["seed-welcome"] = NoteDto("seed-welcome", "Welcome to FirebaseSync", "Came FROM the cloud.", seededAt)
    }

    // override suspend keeps the same "may pause" contract as CloudApi.
    override suspend fun pullSince(since: Long): List<NoteDto> {
        delay(700)                                         // imitate network waiting
        return server.values.filter { it.updatedAt > since }.sortedBy { it.updatedAt }
    }
    override suspend fun push(note: NoteDto): NoteDto {
        delay(500)
        val existing = server[note.id]
        if (existing == null || note.updatedAt >= existing.updatedAt) server[note.id] = note  // server last-write-wins too
        return server.getValue(note.id)
    }
    fun simulateRemoteEdit() { /* add a row as if another device created it */ }
}
      data/CloudApi.kt · FirestoreCloudApi (the REAL cloud)
      // FirestoreCloudApi — talks to a real Google Cloud Firestore database.
// Firebase is initialised in code from 3 constants (no google-services.json needed to build).
class FirestoreCloudApi(context: Context) : CloudApi {

    // Get-or-create the FirebaseApp from the BuildConfig values (fed from local.properties), then get the Firestore instance.
    private val db: FirebaseFirestore = run {
        val app = FirebaseApp.getApps(context).firstOrNull()
            ?: FirebaseApp.initializeApp(context, FirebaseOptions.Builder()
                .setProjectId(FIREBASE_PROJECT_ID)
                .setApplicationId(FIREBASE_APP_ID)
                .setApiKey(FIREBASE_API_KEY)
                .build())
        FirebaseFirestore.getInstance(app)
    }

    // All notes live in one top-level collection; each document's id IS the note id.
    private val notes get() = db.collection("notes")

    override suspend fun pullSince(since: Long): List<NoteDto> {
        // Range query: only documents whose updatedAt is > the pull cursor, oldest-first.
        // .get() returns a Play-services Task; .await() suspends until it resolves (or throws).
        val snapshot = notes
            .whereGreaterThan("updatedAt", since)
            .orderBy("updatedAt", Query.Direction.ASCENDING)
            .get()
            .await()
        // toObject() maps each document's fields onto a NoteDto. NoteDto has default values on
        // every property so Firestore can build it with the no-arg constructor and fill the fields.
        return snapshot.documents.mapNotNull { it.toObject(NoteDto::class.java) }
    }

    override suspend fun push(note: NoteDto): NoteDto {
        // set() creates-or-overwrites the document at notes/{id}. Always wins — no version check.
        notes.document(note.id).set(note).await()
        return note
    }
}

      set() is the push: it writes the full document at notes/{id}, creating it if absent or overwriting it if present. The range query is the pull: whereGreaterThan("updatedAt", since) fetches only documents newer than the cursor. NoteDto must give every property a default value so Firestore can deserialise documents with its no-arg constructor — unlike CloudSync's DTO, no @Serializable annotation is needed here because Firestore does its own object mapping.

      Predict before reveal: why can FakeCloudApi and FirestoreCloudApi be swapped?
      They both implement the same CloudApi interface. The Repository only calls push() and pullSince(), so it does not care whether those functions use a pretend in-memory cloud or a real Firestore database.

      Step 4 · write locally, instantly

      Optimistic writes — Room first, marked Pending

      Every create/edit/delete writes to Room immediately with syncState = PENDING. The user sees the change at once — online or offline — and the row joins the outbox to be pushed later. A delete is a soft-delete so it can still be pushed.

      What the Repository owns The Repository is the traffic director. The DAO only knows the local database. The CloudApi only knows the cloud. The Repository decides the order: save locally first, mark it Pending, send it to the cloud later, then bring cloud changes back into Room.
      data/NoteRepository.kt · the write methods
      class NoteRepository(
    private val dao: NoteDao,
    private val cloud: CloudApi
) {
    val notes: Flow<List<Note>> = dao.observeVisibleNotes()    // what the UI observes

    // suspend because Room writes may pause; the ViewModel calls this from viewModelScope.launch.
    suspend fun addNote(title: String, body: String) {
        dao.upsert(Note(
            id = UUID.randomUUID().toString(),                 // identity before the server sees it
            title = title.trim(), body = body.trim(),
            updatedAt = System.currentTimeMillis(),
            syncState = SyncState.PENDING,                      // joins the outbox
        ))
    }

    suspend fun deleteNote(id: String) {
        val existing = dao.findById(id) ?: return
        dao.upsert(existing.copy(deleted = true,               // SOFT delete (tombstone) — still pushable offline
            updatedAt = System.currentTimeMillis(), syncState = SyncState.PENDING))
    }
}
      Predict before reveal: what should the UI show immediately after an offline delete?
      The note disappears from the visible list, but a hidden tombstone remains in Room with syncState = PENDING. That hidden row is what lets sync upload the delete later.

      Step 5 · reconcile with the cloud

      sync() — upload waiting notes, then download cloud changes

      sync() is the catch-up function. First it uploads local notes that are waiting. Then it downloads newer notes from the cloud. If the network fails, the local notes stay safe in Room and the worker tries again later.

      Repository stepWhat it callsWhat changes
      Save firstdao.upsert(... PENDING)The note appears on screen right away.
      Upload waiting notesdao.notesToPush(), then cloud.push(...)Pending rows go to the cloud. Good uploads become SYNCED.
      Handle failuredao.upsert(... FAILED), then throwThe screen can show a failed badge, and the worker knows to try again.
      Download cloud changesdao.maxUpdatedAt(), then cloud.pullSince(...)New cloud rows are saved into Room, so the UI updates.
      Pick a winnershouldAcceptRemote(local, remote)If two copies changed, FirebaseSync keeps the newer one.
      Remove confirmed deletesdao.hardDelete(id)Only after the cloud receives the tombstone does Room remove the hidden row for good.
      data/NoteRepository.kt · sync()
      suspend fun sync() {
    // suspend because sync waits for Room and cloud calls without blocking the UI thread.
    // 1) PUSH — every row created/edited/deleted offline.
    for (local in dao.notesToPush()) {
        try {
            val serverCopy = cloud.push(local.toDto())     // may throw -> retried by the worker
            if (local.deleted) dao.hardDelete(local.id)    // tombstone confirmed -> remove locally
            else dao.upsert(serverCopy.toEntity(SyncState.SYNCED))
        } catch (e: Exception) {
            dao.upsert(local.copy(syncState = SyncState.FAILED))   // surface it in the UI
            throw e                                                 // let the worker retry
        }
    }
    // 2) PULL — remote rows newer than the newest we already have.
    val since = dao.maxUpdatedAt() ?: 0L
    for (remote in cloud.pullSince(since)) {
        val local = dao.findById(remote.id)
        if (shouldAcceptRemote(local, remote)) {           // last-write-wins
            if (remote.deleted) dao.hardDelete(remote.id)
            else dao.upsert(remote.toEntity(SyncState.SYNCED))
        }
    }
}
      Push before pull Upload your local changes first. Then ask the cloud for new changes. That way your own offline work gets to the cloud before the app downloads the latest cloud copy.
      Predict before reveal: why push before pull?
      The app sends local work first so your offline edits reach the cloud before the app asks for newer cloud data. That keeps local pending work from being accidentally ignored.

      Step 6 · who wins when both changed?

      Conflict resolution — last-write-wins, as a small function

      Sometimes the phone and the cloud both changed the same note. FirebaseSync needs a simple rule. This sample uses last-write-wins: the copy with the newer updatedAt time wins.

      data/NoteRepository.kt · shouldAcceptRemote
      /**
 * Accept the cloud row only if it will not erase a local edit still waiting to upload.
 * If local work is still PENDING, keep it local and push it on the next sync.
 */
fun shouldAcceptRemote(local: Note?, remote: NoteDto): Boolean =
    local == null || (local.syncState == SyncState.SYNCED && remote.updatedAt >= local.updatedAt)
      StrategyHowUsed here?
      Last-write-winsCompare updatedAt; newer overwrites✓ simple, fine for personal data
      Version / optimistic lockServer rejects a stale version; the app downloads the latest copy and tries again.— for shared or multi-user data
      Field-level merge / CRDTMerge non-conflicting fields instead of picking one whole note. CRDT means "conflict-free replicated data type," an advanced structure for collaborative editing.— for collaborative editing
      Predict before reveal: when should FirebaseSync reject a cloud row?
      It should reject a cloud row when the local row is still PENDING. That local edit has not uploaded yet, so replacing it would erase offline work.

      Step 7 · where offline "just queues"

      The background worker — sync when the screen is gone

      SyncWorker runs sync in the background. It is useful because the screen might close, the app might restart, or the network might be offline. WorkManager can wait until the network is back, then run the sync job.

      What the worker does
      • Runs outside the screen, so sync can continue after the user leaves the page.
      • Extends CoroutineWorker, a WorkManager worker type that can call suspend functions.
      • Gets the Repository.
      • Calls repo.sync().
      • Reports success only after upload and download finish.
      What WorkManager adds
      • NetworkType.CONNECTED means "run only when online."
      • Result.retry() means "try again later."
      • BackoffPolicy.EXPONENTIAL means "wait longer between repeated failures."
      • enqueueUniqueWork(...) means "do not start a bunch of duplicate sync jobs."

      ContextAn Android handle to app-level services and resources.The worker uses it to get the same Repository the app uses.
      WorkerParametersInformation WorkManager passes into a worker when it starts it.SyncWorker receives these because every WorkManager worker does.
      doWork()The function WorkManager runs in the background.FirebaseSync calls repo.sync() from here.
      Result.success()The worker reports that the job finished.Used only after upload and download complete.
      ExistingWorkPolicy.APPEND_OR_REPLACEHow WorkManager handles a new request with the same job name.FirebaseSync avoids piling up duplicate sync jobs.
      OneTimeWorkRequestBuilderA builder for one background job request.FirebaseSync creates one sync request each time it asks WorkManager to sync.

      sync/SyncWorker.kt
      class SyncWorker(appContext: Context, params: WorkerParameters) : CoroutineWorker(appContext, params) {
    override suspend fun doWork(): Result {
        // WorkManager runs doWork() in a coroutine, so it can call suspend sync code.
        val repo = NoteRepository.get(applicationContext)   // the SAME repo the UI uses
        return try { repo.sync(); Result.success() }
        catch (e: Exception) { Result.retry() }             // backoff; try again later
    }
    companion object { const val NAME = "cloud-sync" }
}
      sync/SyncScheduler.kt · the constraint that makes offline queue
      fun requestSync(context: Context) {
    val request = OneTimeWorkRequestBuilder<SyncWorker>()
        .setConstraints(Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)  // run when online, DEFER when offline
            .build())
        .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
        .build()
    WorkManager.getInstance(context)
        .enqueueUniqueWork(SyncWorker.NAME, ExistingWorkPolicy.APPEND_OR_REPLACE, request)
}
      This one constraint is the whole offline story With NetworkType.CONNECTED, a sync job waits while the device is offline. When the network comes back, WorkManager runs it. The app does not lose the local notes.
      Predict before reveal: what happens if the sync job is requested with no network?
      WorkManager keeps the job waiting because of NetworkType.CONNECTED. The local Room rows stay safe, and the job runs when the device is online again.

      Step 8 · ViewModel & screen

      Local state in, intents out

      The DAO gives the Repository regular Flows. The ViewModel is the screen's data coordinator: it turns those streams into StateFlows so the screen always has a current value to draw. Every button press still does the same two steps: write to Room, then request a sync.

      Deep dive · the SDK under the hood

      Every Firestore API call, explained

      FirestoreCloudApi implements the two sync operations — push and pull — using eleven distinct Firestore / Firebase SDK calls. The table below maps each call to what it does and where it appears in data/CloudApi.kt. Together they represent the complete surface of the Firestore SDK that this project uses.

      push is the write path: one note → one document. The path is FirebaseOptions → FirebaseApp → FirebaseFirestore → collection() → document() → set() → await().
      pull is the read path: a range query for newer notes. The path is collection() → whereGreaterThan() → orderBy() → get() → await() → toObject().

      API call What it does Where in the code
      FirebaseOptions.Builder()
      .setProjectId()
      .setApplicationId()
      .setApiKey()
      .build()      		 Builds the in-code configuration object that identifies the Firebase project — the programmatic equivalent of google-services.json. setProjectId names the Cloud project, setApplicationId names the registered Android app within it, and setApiKey authenticates SDK calls to Google's backend. build() produces an immutable FirebaseOptions value object. data/CloudApi.kt — FirestoreCloudApi initialiser (private val db)
      FirebaseApp.getApps(context) / FirebaseApp.initializeApp(context, options) getApps() returns every FirebaseApp already registered in this process. We use firstOrNull() to get it if it exists; initializeApp() creates it if not. Firebase is a process-wide singleton — calling initializeApp() twice would throw, so the get-or-create pattern is required. data/CloudApi.kt — FirestoreCloudApi initialiser
      FirebaseFirestore.getInstance(app) Returns the FirebaseFirestore client handle bound to a specific FirebaseApp. This is the root object all queries and writes are issued through. The network connection is lazy — it only opens when the first real read or write is made. data/CloudApi.kt — FirestoreCloudApi initialiser; stored in private val db
      db.collection("notes") Returns a CollectionReference — a lightweight path handle pointing at the top-level notes collection in Firestore. No network call; purely a descriptor. Every note document lives at notes/{id}. data/CloudApi.kt — private val notes property; used by both push and pullSince
      notes.document(id) Returns a DocumentReference at the path notes/{id}. Again, no network call. Using the note's own client-generated UUID as the document id means no server round-trip is needed to assign an id — the same UUID created offline is the Firestore key. data/CloudApi.kt — push()
      .set(note) Creates the document if it does not exist, or completely overwrites it if it does. There is no version check — set() always wins. Returns a Task<Void> (Play-services async type) that resolves when the server confirms the write. Client-side conflict resolution (shouldAcceptRemote) handles the last-write-wins logic; set() itself is unconditional. data/CloudApi.kt — push()
      .whereGreaterThan("updatedAt", since) Adds a range filter to the query: only return documents where the updatedAt field is strictly greater than since (the pull cursor — the epoch-millis of the newest local note). Field names are strings and must match the property names on NoteDto exactly. data/CloudApi.kt — pullSince()
      .orderBy("updatedAt", Query.Direction.ASCENDING) Sorts results oldest-first so the sync engine processes them in chronological order. When a range filter (whereGreaterThan) and orderBy target the same field, Firestore satisfies both with a single auto-created single-field index — no composite index is needed in the Firebase console. data/CloudApi.kt — pullSince()
      .get() Executes the query against Firestore and returns a Task<QuerySnapshot>. The Task type is from Google Play services — it is callback-based, not a Kotlin coroutine. Nothing happens until await() is called on it. data/CloudApi.kt — pullSince()
      .await() From kotlinx-coroutines-play-services (the bridge dependency in build.gradle.kts). Suspends the current coroutine until the Play-services Task completes, then returns the result — or throws if the task failed (network error, permission denied, etc.). This is what makes sequential-looking coroutine code possible instead of nested callbacks. Used for both the get() query and the set() write. data/CloudApi.kt — pullSince() (after .get()) and push() (after .set())

      }      		 toObject() is Firestore's reflective deserialiser: it calls NoteDto's no-arg constructor (which Kotlin generates automatically when every property has a default value) and then sets each field by matching document field names to property names.       implementation(libs.kotlinx.coroutines.play.services)
      and imported at the top of CloudApi.kt:
      import kotlinx.coroutines.tasks.await
      Predict before reveal: why does NoteDto need default values on every property?
      Firestore's toObject() deserialises a document by calling the class's no-arg constructor and then setting fields by name. A Kotlin data class only gets a generated no-arg constructor when all of its parameters have defaults. Without them, toObject() cannot instantiate the class and throws at runtime. This is a Firestore-specific requirement — contrast with CloudSync's @Serializable DTOs, which use a completely different (kotlinx.serialization) mechanism.

      Sample solutions
      1. You tap Add while offline. What are the first three layers touched?

      UI → ViewModel → Repository. The UI sends the user intent, the ViewModel calls the repository from a coroutine, and the Repository writes the note to Room first.

      2. Which Room fields prove the note still needs cloud work?

      syncState = PENDING proves the note is in the outbox. If it is a delete, deleted = true means the row is a tombstone that still needs to be pushed to the cloud.

      3. Why is a delete saved as a tombstone before hardDelete?

      If the row vanished immediately while offline, the app would forget that the delete ever happened. A tombstone keeps enough data to upload the delete later; hardDelete waits until the cloud confirms it.

      4. When the network returns, what does WorkManager do first?

      WorkManager runs SyncWorker.doWork(). The worker calls repo.sync(), which pushes local outbox rows first, saves successful rows back as SYNCED, then pulls newer cloud rows into Room.

      5. A cloud note arrives. How does the UI learn about it?

      The Repository saves the cloud note into Room. Room reruns the DAO query and the Flow emits a new list. The ViewModel's StateFlow holds that latest list, and Compose redraws from it.

      6. What single rule prevents the UI from breaking offline?

      The UI never reads the network directly. It reads and writes Room; the Repository, SyncWorker, and CloudApi make the cloud catch up later.
