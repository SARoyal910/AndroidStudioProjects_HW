Networking, one request at a time

How a note is fetched

Network Parsing is a small Android app that does one thing: it asks for notes, receives JSON, turns that JSON into Kotlin objects, and shows either a spinner, a list, or an error. This page follows that one fetch slowly. Each section says what the code does in plain English first, then shows the real Kotlin.

1. The screen asks

The UI does not fetch by itself. It calls the ViewModel and watches one value: uiState.

2. Data travels down

The ViewModel asks the Repository. The Repository uses either the fake JSON or the real Retrofit API.

3. State travels back up

JSON becomes NoteDto, then Note, then Success(notes). A failure becomes Error(message).

Retrofit kotlinx.serialization ViewModel + StateFlow Loading / Success / Error Kotlin
The one sentence to hold onto The screen only looks at one state value. If that value is Loading, it draws a spinner. If it is Success(notes), it draws the list. If it is Error(message), it draws the message and a Retry button.

Runs offline by default

The app ships in fake mode (provideNoteRepository(useFake = true)): it parses a hardcoded JSON string after a short delay, so it builds, runs, and unit-tests with no network. One flag flips it to the live API at https://jsonplaceholder.typicode.com/posts.

Vocabulary before the code

These words come up on almost every line. Read this table first, then the code sections will feel much less mysterious.

JSON Plain text data from the server. It looks like objects with keys such as "id" and "title". The live API and FAKE_NOTES_JSON both provide JSON shaped like /posts.
NoteDto The network-shaped Kotlin class. Its property names and types must match the JSON that is being decoded. Note.kt, marked with @Serializable.

The shape of the trip

The round trip a request takes

The screen asks for notes; the request travels down through the layers and out to the network (the request path). The JSON that comes back is parsed and mapped, and the result travels back up as a new StateFlow value (the return trip, in green).

↓  viewModel.loadNotes()
↓  repository.getNotes()  (suspend)
↓  api.fetchNotes()
↓  HTTP GET /posts
↑  the return trip: JSON → List<NoteDto> → .map { toNote() } → Success(notes) → recompose

Initialization · before any request

Step 0 — wiring networking on

Three libraries do the work: Retrofit builds the HTTP client, OkHttp is the engine under it, and kotlinx.serialization decodes JSON. The serialization compiler plugin generates the serializer for @Serializable NoteDto.

build.gradle.kts · plugins + networking deps
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.compose)
    // Generates serializers for @Serializable NoteDto.
    alias(libs.plugins.kotlin.serialization)
}

dependencies {
    // kotlinx.serialization: maps JSON <-> @Serializable NoteDto.
    implementation(libs.kotlinx.serialization.json)
    // Retrofit: builds a type-safe HTTP client from the NoteApi interface.
    implementation(libs.retrofit)
    // Lets Retrofit decode response bodies with kotlinx.serialization.
    implementation(libs.retrofit.converter.kotlinx.serialization)
    // OkHttp: the HTTP engine + MediaType used when wiring the converter factory.
    implementation(libs.okhttp)
}
Line / dependencyWhy it is neededWhat would break without it
kotlin.serialization plugin Generates the serializer code for @Serializable NoteDto. The app could not automatically decode JSON into NoteDto.
kotlinx.serialization.json Provides the Json parser and settings such as ignoreUnknownKeys. There would be no JSON engine for the fake parser or Retrofit converter.
retrofit Turns NoteApi into a working HTTP client. @GET("posts") would just be an annotation with no runtime implementation.
retrofit.converter.kotlinx.serialization Connects Retrofit response bodies to the kotlinx.serialization parser. Retrofit could fetch text, but it would not know how to produce List<NoteDto>.
okhttp The HTTP engine Retrofit uses underneath. The app would be missing the low-level client pieces Retrofit relies on.
The INTERNET permission AndroidManifest.xml declares <uses-permission android:name="android.permission.INTERNET" /> actively, so flipping to the real API works with no manifest edit. INTERNET is a normal (install-time) permission — there is no runtime request — and it's harmless while the app runs in its default offline fake mode.

Step 1 · the parsing boundary

DTO vs domain model — two shapes, one mapper

Keep the shape the network speaks (the DTO) separate from the shape your app speaks (the domain model), and convert between them in exactly one place. JSON text arrives, kotlinx.serialization turns it into a NoteDto, and a mapper turns that into a clean Note.

Note.kt · the DTO (network shape)
@Serializable                                  // <-- makes this class JSON-decodable
data class NoteDto(
    val userId: Int,                           // matches JSON "userId" (parsed, then dropped)
    val id: Int,                               // matches JSON "id"
    val title: String,                         // matches JSON "title"
    val body: String,                          // matches JSON "body"
)
Note.kt · the domain model + the mapper
// A PLAIN data class: no @Serializable, no knowledge of JSON, and NO userId.
data class Note(
    val id: Int,
    val title: String,
    val body: String,
)

// The SINGLE place network-shape becomes app-shape.
fun NoteDto.toNote(): Note =                   // extension fn: reads like dto.toNote()
    Note(id = id, title = title, body = body)  // userId is intentionally ignored
Why two shapes The UI and ViewModel never depend on the API's exact wire format. If the server renamed a field or added twenty more, only NoteDto and toNote() would change — not the screens or the ViewModel. The mapper is the one testable seam between "network shape" and "app shape".
PieceWhat the student should noticeReason
@Serializable It is on NoteDto, not on Note. Only the network shape needs JSON decoding. The domain model can stay plain and UI-focused.
userId It exists in NoteDto but not in Note. The JSON includes it, so the DTO parses it. The UI does not display it, so the domain model drops it.
id, title, body They exist in both classes with compatible types. These are the fields the UI actually needs to show each row.
fun NoteDto.toNote() This is an extension function, so it reads like dto.toNote(). The mapping code lives in one named place instead of being repeated wherever notes are fetched.
No @SerialName The property names already match the JSON keys exactly. If the server used a different key, such as "user_id", then @SerialName("user_id") would be needed.

Step 2 · the endpoint, as an interface

The Retrofit API — describe it, don't write it

With Retrofit you describe the endpoint as an annotated interface method; Retrofit generates the actual HTTP code at runtime. You never write sockets, hand-built URLs, or manual JSON parsing — you declare "GET /posts returns a List<NoteDto>" and call a suspend function.

NoteApi.kt · the interface
interface NoteApi {
    // GET {baseUrl}/posts -> a JSON array, decoded into List<NoteDto>.
    // suspend: Retrofit runs the network I/O off the main thread and resumes you with the parsed list.
    @GET("posts")
    suspend fun fetchNotes(): List<NoteDto>
}
NoteApi.kt · assembling Retrofit
// ignoreUnknownKeys = true -> if the server adds NEW fields we didn't model, decoding
// keeps working instead of throwing. The #1 real-world robustness setting.
private val json = Json { ignoreUnknownKeys = true }

fun provideNoteApi(): NoteApi =
    Retrofit.Builder()
        .baseUrl("https://jsonplaceholder.typicode.com/")   // every @GET path is appended here; MUST end in "/"
        .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
        .build()
        .create(NoteApi::class.java)                         // Retrofit generates the implementation
Two robustness details The baseUrl must end in / (Retrofit appends the relative path), and ignoreUnknownKeys = true means a server that grows new fields won't break your parser.
CodeRead it asImportant detail
interface NoteApi A description of the server calls the app knows how to make. There is no method body because Retrofit creates the implementation at runtime.
@GET("posts") Make an HTTP GET request to {baseUrl}/posts. The path is relative. It is appended to https://jsonplaceholder.typicode.com/.
suspend fun fetchNotes() This function must be called from a coroutine and can pause while waiting for I/O. Suspending is not blocking. The UI thread is not frozen while the request is in progress.
List<NoteDto> The response body should be a JSON array, and each JSON object should decode into one NoteDto. If the JSON is malformed or has the wrong type for a required field, decoding throws.
.create(NoteApi::class.java) Ask Retrofit to generate an object that implements NoteApi. This is the moment the interface becomes something callable.

Step 3 · one source of truth

The repository — Real, Fake, and the switch

The rest of the app should not care where notes come from. It asks a NoteRepository for notes; behind that interface sits either a Real implementation (hits the network and maps the DTOs) or a Fake one (parses a hardcoded JSON string after a small delay). Swapping them is a one-line change — which is what lets the app and the tests run completely offline.

NoteRepository.kt · contract + real + fake + switch
interface NoteRepository {
    // suspend because the real impl does network I/O; may THROW on failure.
    suspend fun getNotes(): List<Note>
}

class RealNoteRepository(private val api: NoteApi = provideNoteApi()) : NoteRepository {
    override suspend fun getNotes(): List<Note> {
        val dtos: List<NoteDto> = api.fetchNotes()   // network + parse (throws if offline)
        return dtos.map { it.toNote() }              // DTO -> domain, the parsing payoff
    }
}

class FakeNoteRepository : NoteRepository {
    private val json = Json { ignoreUnknownKeys = true }
    override suspend fun getNotes(): List<Note> {
        delay(600)                                   // imitate latency so Loading is visible
        val dtos: List<NoteDto> = json.decodeFromString(FAKE_NOTES_JSON)  // still parses JSON!
        return dtos.map { it.toNote() }
    }
}

// THE SWITCH — change one argument to toggle offline (default) vs live.
fun provideNoteRepository(useFake: Boolean = true): NoteRepository =
    if (useFake) FakeNoteRepository() else RealNoteRepository()
The fake still parses real JSON Rather than building NoteDto objects by hand, the fake decodeFromStrings a hardcoded JSON payload shaped exactly like the real response — so it exercises the same parse step, just without the network. That's why the offline tests genuinely test parsing.
Repository pieceJobWhat the ViewModel sees
NoteRepository The contract: "I can return a List<Note>, and I may suspend while doing it." Only the interface. The ViewModel does not know whether the implementation is fake or real.
RealNoteRepository Calls api.fetchNotes(), receives List<NoteDto>, maps it to List<Note>. A suspending call that either returns notes or throws.
FakeNoteRepository Waits delay(600), decodes FAKE_NOTES_JSON, maps it to List<Note>. The same suspending call shape as real mode, but deterministic and offline.
provideNoteRepository(useFake = true) The switch that chooses fake by default. The ViewModel receives whichever implementation this factory returns.
delay(600) A non-blocking pause in the fake repository. The loading state is visible, but the main thread is not blocked.

Step 4 · the screen has exactly three states

The UiState pattern — a sealed interface

A network screen is never just "the data". At any moment it is in one of three states — still Loading, loaded Success (with data), or failed Error (with a message). Model those with a sealed interface, and the compiler knows the complete set, so a when over it is exhaustive — you physically cannot forget a case.

NotesViewModel.kt · NotesUiState
sealed interface NotesUiState {
    /** The request is in flight; show a spinner. (A singleton — no data to carry.) */
    data object Loading : NotesUiState

    /** The request succeeded. */
    data class Success(val notes: List<Note>) : NotesUiState

    /** The request failed; carry a human-readable reason. */
    data class Error(val message: String) : NotesUiState
}
StateWhat it carriesWhat the screen must drawWhen it happens
Loading Nothing. It is a singleton object. A spinner and loading text. At startup and every time Retry calls loadNotes().
Success(notes) The parsed list of clean domain Note objects. A scrolling list of note titles and bodies. After the repository returns notes without throwing.
Error(message) A human-readable message from the caught exception. The error title, the message, and a Retry button. After no network, timeout, malformed JSON, HTTP failure, or any other exception.

Step 5 · the conductor

The ViewModel — load, and turn failures into a state

The ViewModel exposes a private MutableStateFlow as a public read-only StateFlow, kicks off the first load in init, and wraps the suspending call in a single try/catch — the one place any failure becomes a visible Error state instead of a crash.

NotesViewModel.kt · state + loadNotes()
class NotesViewModel(
    private val repository: NoteRepository = provideNoteRepository(),   // injected; defaults to offline fake
) : ViewModel() {

    // private mutable -> public read-only: the standard one-way exposure.
    private val _uiState = MutableStateFlow<NotesUiState>(NotesUiState.Loading)
    val uiState: StateFlow<NotesUiState> = _uiState.asStateFlow()

    init { loadNotes() }                                   // start fetching immediately

    fun loadNotes() {
        viewModelScope.launch {
            _uiState.value = NotesUiState.Loading           // 1) show the spinner
            try {
                val notes = repository.getNotes()           // 2) suspend: network/parse
                _uiState.value = NotesUiState.Success(notes) // 3a) publish the data
            } catch (e: Exception) {
                _uiState.value = NotesUiState.Error(         // 3b) ANY failure -> Error state
                    e.message ?: "Something went wrong while loading notes."
                )
            }
        }
    }
}
Why viewModelScope The coroutine is tied to the ViewModel's lifetime, so it's cancelled if the ViewModel is cleared, and the suspending getNotes() never blocks the main thread. loadNotes() is safe to call again — that's exactly what the Retry button does.
Line / patternMeaningWhy it matters
private val repository The ViewModel receives a data source instead of creating one directly inside loadNotes(). This keeps the ViewModel focused on state, and it makes tests or fake data easier.
MutableStateFlow(Loading) The first state value is Loading. The screen has a valid state immediately, before the repository finishes.
asStateFlow() Expose read-only state to the UI. The screen can observe state but cannot accidentally change it.
init { loadNotes() } Start the first fetch when the ViewModel is created. The user does not need to tap anything to see notes load on app launch.
_uiState.value = Loading Publish the loading state before work starts. Retry visibly resets the screen instead of leaving the old error or old list in place.
try { ... } catch (e: Exception) One boundary converts thrown failures into UI state. Network and parse failures become a controlled error screen, not an app crash.

Step 6 · where it starts and ends on screen

The screen — collect the state, render it exhaustively

The stateful NotesScreen obtains the ViewModel and observes its StateFlow; the stateless NotesContent draws whichever state is current. Splitting "get the state" from "draw the state" is what makes the screen previewable without a real ViewModel.

MainActivity.kt · NotesScreen (stateful)
@Composable
fun NotesScreen(modifier: Modifier = Modifier, viewModel: NotesViewModel = viewModel()) {
    // Subscribe to the StateFlow lifecycle-awarely (pauses while off-screen).
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()
    NotesContent(uiState = uiState, onRetry = viewModel::loadNotes, modifier = modifier)
}
MainActivity.kt · NotesContent (stateless) — one exhaustive when
@Composable
fun NotesContent(uiState: NotesUiState, onRetry: () -> Unit, modifier: Modifier = Modifier) {
    when (uiState) {                                       // exhaustive: sealed type, no else needed
        is NotesUiState.Loading -> LoadingState(modifier)            // a centered spinner
        is NotesUiState.Success -> NotesList(uiState.notes, modifier) // a LazyColumn of notes
        is NotesUiState.Error   -> ErrorState(uiState.message, onRetry, modifier) // message + Retry
    }
}
Previews drive the stateless half The @Previews pass hand-made states (Success(previewNotes), Loading, Error("No internet connection.")) straight into NotesContent — never a real ViewModel, which would try to run a coroutine the design pane can't execute.
UI pieceResponsibilityWhy it is split this way
NotesScreen() Gets the ViewModel, collects uiState, passes state and Retry callback down. This is the stateful wrapper. It knows where state comes from.
NotesContent() Receives a NotesUiState and draws it with an exhaustive when. This is stateless and easy to preview because it does not create the ViewModel.
collectAsStateWithLifecycle() Subscribes the composable to the ViewModel's StateFlow. When the flow emits a new state, Compose recomposes the UI safely with lifecycle awareness.
onRetry = viewModel::loadNotes Passes the ViewModel function itself as a callback. The Error UI can retry without knowing how loading works internally.
when (uiState) Chooses Loading, Success, or Error UI. Because NotesUiState is sealed, missing a state becomes a compiler problem.

Step through the exact chain the code follows on a successful load. The first steps are the request path; the last two are the return trip that updates the screen.

When things go wrong

Error handling — one catch, one Error state

A real network call can fail in many ways. The single try/catch in loadNotes() turns any of them into one Error(message) state with a Retry button — never a crash.

What goes wrongThe thrown failureWhat the user sees
No network / airplane modeUnknownHostException“Couldn't load notes” + the message + Retry
Slow server / timeoutSocketTimeoutExceptionSame Error state; Retry re-runs the load
Malformed / unexpected JSONSerializationExceptionSame Error state (a parse failure, not a crash)
HTTP 4xx / 5xxHttpExceptionSame Error state with the status message
Retry is just loadNotes() again The error screen's button is wired to viewModel::loadNotes. Because loadNotes() resets to Loading and re-enters the same try/catch, recovering from a transient failure costs one tap and zero extra code.
ignoreUnknownKeys handles the friendly case Note that a server merely adding fields does not land here — Json { ignoreUnknownKeys = true } decodes successfully and the extra fields are dropped. Only genuinely malformed or type-mismatched JSON throws.

Clear up the likely questions

Common questions, answered directly

These are the questions students usually ask after reading the code. Each answer points back to the exact responsibility in the app.

Why do we need both NoteDto and Note?

NoteDto belongs to the server shape; Note belongs to the app shape. Keeping them separate means the UI and ViewModel do not depend on every server field. The mapper is the one controlled place where network data becomes app data.

Why parse userId if the app drops it?

The JSON contains userId, so NoteDto models it honestly. The UI does not use it, so toNote() leaves it behind. This shows that parsing the response and designing the UI model are related but not identical.

Where is the real HTTP request code?

You do not write it by hand. @GET("posts") describes the request, and Retrofit.Builder().create(NoteApi::class.java) generates an implementation of that interface. Calling api.fetchNotes() is what triggers the generated HTTP GET.

Why is fetchNotes() marked suspend?

Network I/O takes time. suspend lets the coroutine pause until the response is ready without blocking the main thread. The user can still see a responsive Loading UI while the work is in progress.

Why does fake mode still decode JSON?

The fake repository is offline, but it is not skipping the parsing lesson. It decodes FAKE_NOTES_JSON into List<NoteDto> and maps those DTOs to Note, so the same parse-and-map idea is exercised without depending on the internet.

What actually causes the screen to change from Loading to Success?

The ViewModel assigns _uiState.value = NotesUiState.Success(notes). The public uiState emits that new value, collectAsStateWithLifecycle() sees it, and Compose recomposes NotesContent(), which now takes the Success branch.

Why is there no manual refresh call on the list?

Compose UI is declarative. The screen describes what to draw for each state. When the state value changes, Compose reruns the relevant composables. You update the state, not the list widget.

What happens if the server adds a new field?

Because the JSON parser is configured with ignoreUnknownKeys = true, decoding still succeeds. The unknown field is ignored unless you later add it to NoteDto and map it into the domain model.

What happens if the server sends the wrong type?

A wrong type, such as "title": 404 when title must be a String, is not safe to ignore. Decoding throws a serialization exception. The ViewModel catches that exception and publishes Error(message).

Why is NotesUiState sealed?

A sealed type gives the compiler the complete list of possible states. That makes the when (uiState) in NotesContent() exhaustive, so adding or missing a state becomes visible during development.

Why do previews pass fake states instead of creating a ViewModel?

Android Studio previews need quick, deterministic UI. A real ViewModel would start coroutines and use a repository. Passing literal states like Success(previewNotes) previews the UI without network, timers, or lifecycle complexity.

Where should new behavior go if the app grows?

New response fields start in NoteDto and toNote(). New endpoints go in NoteApi. New data-source choices go behind NoteRepository. New screen states go in NotesUiState and must be rendered in NotesContent().

Debug faster

Common mistakes and exactly what they look like

These are the failure patterns students are most likely to hit. Match the symptom to the cause, then inspect the named file.

Missing @Serializable

Symptom: JSON decoding cannot find a serializer for NoteDto.

Fix: Keep @Serializable on DTO classes and keep the serialization plugin enabled.

Wrong baseUrl

Symptom: Retrofit complains about the base URL or the request goes to the wrong path.

Fix: Use https://jsonplaceholder.typicode.com/ with the trailing slash.

No INTERNET permission

Symptom: Real mode cannot reach the network even though the device has a connection.

Fix: Keep <uses-permission android:name="android.permission.INTERNET" /> in the manifest.

DTO used directly in UI

Symptom: Server-only fields leak into screen code, and API changes ripple everywhere.

Fix: Map NoteDto to Note in the repository boundary.

Error handled in the UI only

Symptom: The app can crash before Compose gets a chance to show a friendly state.

Fix: Keep the try/catch around repository.getNotes() in the ViewModel.

Preview creates a real ViewModel

Symptom: Android Studio preview is slow, flaky, or tries to run app logic.

Fix: Preview NotesContent() with hand-made NotesUiState values.

Teacher note Turn each mistake into a five-minute debug drill: show the symptom first, then ask which layer owns the fix.

Run & inspect

The app builds and tests offline by default. Build from the NetworkParsing/ project directory:

terminal · from NetworkParsing/
./gradlew assembleDebug --console=plain --stacktrace   # build (offline fake mode)
./gradlew testDebugUnitTest                            # mapper + fake-repository unit tests

Switch to the live API

  • In NoteRepository.kt, change the default to provideNoteRepository(useFake = false).
  • The INTERNET permission is already declared in AndroidManifest.xml, so no manifest edit is needed.
  • Run on a device/emulator with a connection — it will hit https://jsonplaceholder.typicode.com/posts live.
