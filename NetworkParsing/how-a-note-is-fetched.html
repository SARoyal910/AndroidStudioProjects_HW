Networking, one request at a time

How the weather is fetched

Network Parsing is a small Android app that does one thing: it asks the Open-Meteo API for the current weather, receives nested JSON, flattens that JSON into one Kotlin object, and shows either a spinner, a weather card, or an error. This page follows that one fetch slowly. Each section says what the code does in plain English first, then shows the real Kotlin.

1. The screen asks

The UI does not fetch by itself. You tap a city chip; it calls the ViewModel and watches one value: uiState.

2. Data travels down

The ViewModel asks the Repository. The Repository uses either the real Retrofit API or the offline fake JSON.

3. State travels back up

Nested JSON becomes a ForecastResponseDto, then a flat Weather, then Success(weather). A failure becomes Error(message).

Retrofit kotlinx.serialization ViewModel + StateFlow Loading / Success / Error Kotlin
The one sentence to hold onto The screen only looks at one state value. If that value is Loading, it draws a spinner. If it is Success(weather), it draws the weather card. If it is Error(message), it draws the message and a Retry button.

Runs live by default — no key needed

Open-Meteo is free and needs no API key, so the app ships in real mode (provideWeatherRepository(useFake = false)): it hits the live API at https://api.open-meteo.com/v1/forecast the moment you run it. One flag flips it to an offline fake that parses a hardcoded JSON string after a short delay, so it still builds and unit-tests with no network.

Vocabulary before the code

These words come up on almost every line. Read this table first, then the code sections will feel much less mysterious.

TermPlain meaningWhere it appears in this app
JSON Plain text data from the server. Here it is a nested object with a "current" block whose keys are snake_case, such as "temperature_2m". The live API and FAKE_FORECAST_JSON both provide JSON shaped like /v1/forecast.
ForecastResponseDto The network-shaped Kotlin class. It mirrors the whole JSON object, holding a nested CurrentDto. Weather.kt, marked with @Serializable.
CurrentDto The nested "current" object. Each property uses @SerialName to bridge a snake_case JSON key to a camelCase Kotlin name. Weather.kt, e.g. @SerialName("temperature_2m") temperature.
Weather The app-shaped Kotlin class. It is flat and contains only what the UI actually needs. Weather.kt, with temperatureC, description, humidityPercent, … (no nesting, no weather_code as raw text).
toWeather() The mapper. It flattens one nested ForecastResponseDto into one flat Weather. ForecastResponseDto.toWeather(city) is called after Retrofit or the fake repository parses JSON.
Repository A data-source wrapper. The ViewModel asks it for the weather without caring whether it came from the network or the fake. WeatherRepository.kt, with RealWeatherRepository and FakeWeatherRepository.
Retrofit A library that turns an annotated Kotlin interface into real HTTP calls. WeatherApi.kt, especially @GET("v1/forecast") and provideWeatherApi().
StateFlow A stream-like state holder that always has a current value. The UI reads it and redraws when it changes. WeatherViewModel.uiState, collected by WeatherScreen().
Recomposition Compose running a composable again because state changed, so the screen can draw the new result. When _uiState changes from Loading to Success or Error.

The shape of the trip

The round trip a request takes

The screen asks for the weather; the request travels down through the layers and out to the network (the request path). The JSON that comes back is parsed and mapped, and the result travels back up as a new StateFlow value (the return trip, in green).

↓  viewModel.loadWeather(city)
↓  repository.getWeather(city)  (suspend)
↓  api.fetchForecast(lat, lon)
↓  HTTP GET /v1/forecast?latitude=…&longitude=…
↑  the return trip: JSON → ForecastResponseDto → .toWeather(city) → Success(weather) → recompose

Initialization · before any request

Step 0 — wiring networking on

Three libraries do the work: Retrofit builds the HTTP client, OkHttp is the engine under it, and kotlinx.serialization decodes JSON. The serialization compiler plugin generates the serializers for the @Serializable DTOs (ForecastResponseDto + CurrentDto).

build.gradle.kts · plugins + networking deps
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.compose)
    // Generates serializers for the @Serializable DTOs (ForecastResponseDto + CurrentDto).
    alias(libs.plugins.kotlin.serialization)
}

dependencies {
    // kotlinx.serialization: maps JSON <-> the @Serializable DTOs.
    implementation(libs.kotlinx.serialization.json)
    // Retrofit: builds a type-safe HTTP client from the WeatherApi interface.
    implementation(libs.retrofit)
    // Lets Retrofit decode response bodies with kotlinx.serialization.
    implementation(libs.retrofit.converter.kotlinx.serialization)
    // OkHttp: the HTTP engine + MediaType used when wiring the converter factory.
    implementation(libs.okhttp)
}
Line / dependencyWhy it is neededWhat would break without it
kotlin.serialization plugin Generates the serializer code for the @Serializable DTOs. The app could not automatically decode JSON into ForecastResponseDto.
kotlinx.serialization.json Provides the Json parser and settings such as ignoreUnknownKeys. There would be no JSON engine for the fake parser or Retrofit converter.
retrofit Turns WeatherApi into a working HTTP client. @GET("v1/forecast") would just be an annotation with no runtime implementation.

The INTERNET permission AndroidManifest.xml declares <uses-permission android:name="android.permission.INTERNET" /> actively, so flipping to the real API works with no manifest edit. INTERNET is a normal (install-time) permission — there is no runtime request — and it's harmless while the app runs in its default offline fake mode.

Step 1 · the parsing boundary

DTO vs domain model — two shapes, one mapper

Keep the shape the network speaks (the DTOs) separate from the shape your app speaks (the domain model), and convert between them in exactly one place. Nested JSON text arrives, kotlinx.serialization turns it into a ForecastResponseDto, and a mapper flattens that into a clean Weather.

Weather.kt · the nested DTOs (network shape)
@Serializable                                  // <-- makes this class JSON-decodable
data class ForecastResponseDto(
    val current: CurrentDto,                   // matches JSON "current": { … } (a NESTED object)
)

@Serializable
data class CurrentDto(
    val time: String,                                          // "2026-06-25T12:30"
    @SerialName("temperature_2m") val temperature: Double,     // snake_case JSON key -> camelCase Kotlin
    @SerialName("apparent_temperature") val feelsLike: Double, // "feels like", °C
    @SerialName("relative_humidity_2m") val humidity: Int,     // %
    @SerialName("weather_code") val weatherCode: Int,          // a WMO NUMBER (0 = clear), decoded later
    @SerialName("wind_speed_10m") val windSpeed: Double,       // km/h
)
Weather.kt · the flat domain model + the mapper
// A PLAIN, FLAT data class: no @Serializable, no nesting, no cryptic weather_code.
data class Weather(
    val city: String, val country: String,
    val temperatureC: Int, val feelsLikeC: Int,
    val description: String,                   // weather_code turned into human text
    val humidityPercent: Int, val windKmh: Int,
    val weatherCode: Int,                      // raw WMO code kept only to pick an emoji
)

// The SINGLE place the NESTED network-shape becomes the FLAT app-shape.
fun ForecastResponseDto.toWeather(city: City): Weather =      // extension fn: dto.toWeather(city)
    Weather(
        city = city.name, country = city.country,
        temperatureC = current.temperature.roundToInt(),       // reach INTO the nested "current"
        feelsLikeC = current.feelsLike.roundToInt(),
        description = wmoDescription(current.weatherCode),      // 0 -> "clear sky"
        humidityPercent = current.humidity,
        windKmh = current.windSpeed.roundToInt(),
        weatherCode = current.weatherCode,
    )
Why two shapes The UI and ViewModel never depend on the API's exact wire format. If Open-Meteo renamed a field or added twenty more, only the DTOs and toWeather() would change — not the screens or the ViewModel. The mapper is the one testable seam between "network shape" and "app shape".
PieceWhat the student should noticeReason
@Serializable It is on the DTOs, not on Weather. Only the network shape needs JSON decoding. The domain model can stay plain and UI-focused.
Nesting ForecastResponseDto holds a CurrentDto, mirroring the JSON's "current" block. Real APIs return nested objects; the DTOs mirror that nesting so decoding lines up key-for-key.
@SerialName("temperature_2m") Every current key is snake_case on the wire but camelCase in Kotlin. @SerialName bridges each one. Drop it and the decoder cannot find temperature_2m for the temperature property.
weather_code It decodes into an Int, not text — a numeric WMO code. The mapper turns the number into a phrase via wmoDescription(0) → "clear sky"; the UI never sees the raw code as text.
fun ForecastResponseDto.toWeather() This is an extension function, so it reads like dto.toWeather(city). The mapping code lives in one named place; it FLATTENS the nested DTO instead of just copying matching fields.

Step 2 · the endpoint, as an interface

The Retrofit API — describe it, don't write it

With Retrofit you describe the endpoint as an annotated interface method; Retrofit generates the actual HTTP code at runtime. You never write sockets, hand-built URLs, or manual JSON parsing — you declare "GET v1/forecast with these query params returns a ForecastResponseDto" and call a suspend function. Each @Query becomes one ?name=value in the URL.

WeatherApi.kt · the interface
interface WeatherApi {
    // GET {baseUrl}v1/forecast?latitude=…&longitude=…&current=… -> one JSON object, decoded into ForecastResponseDto.
    // suspend: Retrofit runs the network I/O off the main thread and resumes you with the parsed object.
    @GET("v1/forecast")
    suspend fun fetchForecast(
        @Query("latitude") latitude: Double,       // becomes ?latitude=28.5384
        @Query("longitude") longitude: Double,     // becomes &longitude=-81.3789
        @Query("current") current: String =        // becomes &current=temperature_2m,…,wind_speed_10m
            "temperature_2m,relative_humidity_2m,apparent_temperature,weather_code,wind_speed_10m",
    ): ForecastResponseDto
}
WeatherApi.kt · assembling Retrofit
// ignoreUnknownKeys = true -> the real API returns MANY fields we don't model (latitude,
// timezone, current_units, …); ignoring them keeps decoding working. The #1 robustness setting.
private val json = Json { ignoreUnknownKeys = true }

fun provideWeatherApi(): WeatherApi =
    Retrofit.Builder()
        .baseUrl("https://api.open-meteo.com/")             // every @GET path is appended here; MUST end in "/"
        .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
        .build()
        .create(WeatherApi::class.java)                      // Retrofit generates the implementation
Two robustness details The baseUrl must end in / (Retrofit appends the relative path), and ignoreUnknownKeys = true means the many fields we don't model (and any new ones) won't break your parser.
CodeRead it asImportant detail
interface WeatherApi A description of the server calls the app knows how to make. There is no method body because Retrofit creates the implementation at runtime.
@GET("v1/forecast") Make an HTTP GET request to {baseUrl}v1/forecast. The path is relative. It is appended to https://api.open-meteo.com/.
@Query("latitude") … Each annotated parameter becomes one ?name=value in the URL. Open-Meteo is coordinate-based, so the city's latitude/longitude go here; current lists the readings.
suspend fun fetchForecast() This function must be called from a coroutine and can pause while waiting for I/O. Suspending is not blocking. The UI thread is not frozen while the request is in progress.
ForecastResponseDto The response body should be a JSON object that decodes into one nested ForecastResponseDto. If the JSON is malformed or has the wrong type for a required field, decoding throws.
.create(WeatherApi::class.java) Ask Retrofit to generate an object that implements WeatherApi. This is the moment the interface becomes something callable.

Step 3 · one source of truth

The repository — Real, Fake, and the switch

The rest of the app should not care where the weather comes from. It asks a WeatherRepository for a city's weather; behind that interface sits either a Real implementation (hits the network and maps the DTO) or a Fake one (parses a hardcoded JSON string after a small delay). Swapping them is a one-line change — which is what lets the app and the tests run completely offline.

WeatherRepository.kt · contract + real + fake + switch
interface WeatherRepository {
    // suspend because the real impl does network I/O; may THROW on failure.
    suspend fun getWeather(city: City): Weather
}

class RealWeatherRepository(private val api: WeatherApi = provideWeatherApi()) : WeatherRepository {
    override suspend fun getWeather(city: City): Weather {
        val dto: ForecastResponseDto =
            api.fetchForecast(city.latitude, city.longitude)  // network + parse (throws if offline)
        return dto.toWeather(city)                            // nested DTO -> flat domain, the parsing payoff
    }
}

class FakeWeatherRepository : WeatherRepository {
    private val json = Json { ignoreUnknownKeys = true }
    override suspend fun getWeather(city: City): Weather {
        delay(600)                                            // imitate latency so Loading is visible
        val dto: ForecastResponseDto = json.decodeFromString(FAKE_FORECAST_JSON)  // still parses JSON!
        return dto.toWeather(city)
    }
}

// THE SWITCH — change one argument to toggle live (default) vs offline.
fun provideWeatherRepository(useFake: Boolean = false): WeatherRepository =
    if (useFake) FakeWeatherRepository() else RealWeatherRepository()
The fake still parses real JSON Rather than building a ForecastResponseDto by hand, the fake decodeFromStrings a hardcoded JSON payload shaped exactly like the real nested response — so it exercises the same parse-and-flatten step, just without the network. That's why the offline tests genuinely test parsing.
Repository pieceJobWhat the ViewModel sees
WeatherRepository The contract: "I can return a Weather for a city, and I may suspend while doing it." Only the interface. The ViewModel does not know whether the implementation is fake or real.
RealWeatherRepository Calls api.fetchForecast(...), receives a ForecastResponseDto, flattens it to Weather. A suspending call that either returns the weather or throws.
FakeWeatherRepository Waits delay(600), decodes FAKE_FORECAST_JSON, flattens it to Weather. The same suspending call shape as real mode, but deterministic and offline.
provideWeatherRepository(useFake = false) The switch that chooses the live, keyless real repository by default. The ViewModel receives whichever implementation this factory returns.
delay(600) A non-blocking pause in the fake repository. The loading state is visible, but the main thread is not blocked.

Step 4 · the screen has exactly three states

The UiState pattern — a sealed interface

A network screen is never just "the data". At any moment it is in one of three states — still Loading, loaded Success (with the weather), or failed Error (with a message). Model those with a sealed interface, and the compiler knows the complete set, so a when over it is exhaustive — you physically cannot forget a case.

WeatherViewModel.kt · WeatherUiState
sealed interface WeatherUiState {
    /** The request is in flight; show a spinner. (A singleton — no data to carry.) */
    data object Loading : WeatherUiState

    /** The request succeeded. */
    data class Success(val weather: Weather) : WeatherUiState

    /** The request failed; carry a human-readable reason. */
    data class Error(val message: String) : WeatherUiState
}
StateWhat it carriesWhat the screen must drawWhen it happens
Loading Nothing. It is a singleton object. A spinner and loading text. At startup, on each city tap, and every time Retry calls loadWeather().
Success(weather) The parsed, flat domain Weather object. A weather card: city, temperature, condition, feels-like, humidity, wind. After the repository returns the weather without throwing.
Error(message) A human-readable message from the caught exception. The error title, the message, and a Retry button. After no network, timeout, malformed JSON, HTTP failure, or any other exception.

Step 5 · the conductor

The ViewModel — load, and turn failures into a state

The ViewModel exposes a private MutableStateFlow as a public read-only StateFlow, kicks off the first load in init, and wraps the suspending call in a single try/catch — the one place any failure becomes a visible Error state instead of a crash.

WeatherViewModel.kt · state + loadWeather()
class WeatherViewModel(
    private val repository: WeatherRepository = provideWeatherRepository(),  // injected; defaults to the live API
) : ViewModel() {

    var city: City = PRESET_CITIES.first()                 // the city shown; tapping a chip changes it
        private set

    // private mutable -> public read-only: the standard one-way exposure.
    private val _uiState = MutableStateFlow<WeatherUiState>(WeatherUiState.Loading)
    val uiState: StateFlow<WeatherUiState> = _uiState.asStateFlow()

    init { loadWeather() }                                  // start fetching immediately

    fun loadWeather(city: City = this.city) {
        this.city = city
        viewModelScope.launch {
            _uiState.value = WeatherUiState.Loading             // 1) show the spinner
            try {
                val weather = repository.getWeather(city)        // 2) suspend: network/parse
                _uiState.value = WeatherUiState.Success(weather) // 3a) publish the data
            } catch (e: Exception) {
                _uiState.value = WeatherUiState.Error(           // 3b) ANY failure -> Error state
                    e.message ?: "Something went wrong while loading the weather."
                )
            }
        }
    }
}
Why viewModelScope The coroutine is tied to the ViewModel's lifetime, so it's cancelled if the ViewModel is cleared, and the suspending getWeather() never blocks the main thread. loadWeather() is safe to call again — that's exactly what tapping another city chip or the Retry button does.
Line / patternMeaningWhy it matters
private val repository The ViewModel receives a data source instead of creating one directly inside loadWeather(). This keeps the ViewModel focused on state, and it makes tests or fake data easier.
MutableStateFlow(Loading) The first state value is Loading. The screen has a valid state immediately, before the repository finishes.
asStateFlow() Expose read-only state to the UI. The screen can observe state but cannot accidentally change it.
init { loadWeather() } Start the first fetch when the ViewModel is created. The user does not need to tap anything to see the weather load on app launch.
_uiState.value = Loading Publish the loading state before work starts. A city tap or Retry visibly resets the screen instead of leaving the old error or old card in place.
try { ... } catch (e: Exception) One boundary converts thrown failures into UI state. Network and parse failures become a controlled error screen, not an app crash.

Step 6 · where it starts and ends on screen

The screen — collect the state, render it exhaustively

The stateful WeatherScreen obtains the ViewModel, observes its StateFlow, and shows a row of city chips; the stateless WeatherContent draws whichever state is current. Splitting "get the state" from "draw the state" is what makes the screen previewable without a real ViewModel.

MainActivity.kt · WeatherScreen (stateful)
@Composable
fun WeatherScreen(modifier: Modifier = Modifier, viewModel: WeatherViewModel = viewModel()) {
    // Subscribe to the StateFlow lifecycle-awarely (pauses while off-screen).
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()
    Column(modifier = modifier.fillMaxSize()) {
        CityChips(selected = viewModel.city, onSelect = { city -> viewModel.loadWeather(city) })
        WeatherContent(uiState = uiState, onRetry = { viewModel.loadWeather() })
    }
}
MainActivity.kt · WeatherContent (stateless) — one exhaustive when
@Composable
fun WeatherContent(uiState: WeatherUiState, onRetry: () -> Unit, modifier: Modifier = Modifier) {
    when (uiState) {                                       // exhaustive: sealed type, no else needed
        is WeatherUiState.Loading -> LoadingState(modifier)                  // a centered spinner
        is WeatherUiState.Success -> WeatherCard(uiState.weather, modifier)   // the weather card
        is WeatherUiState.Error   -> ErrorState(uiState.message, onRetry, modifier) // message + Retry
    }
}
Previews drive the stateless half The @Previews pass hand-made states (Success(previewWeather), Loading, Error("Unable to resolve host \"api.open-meteo.com\"")) straight into WeatherContent — never a real ViewModel, which would try to run a coroutine the design pane can't execute.
UI pieceResponsibilityWhy it is split this way
WeatherScreen() Gets the ViewModel, collects uiState, shows the city chips, passes state and Retry callback down. This is the stateful wrapper. It knows where state comes from.
WeatherContent() Receives a WeatherUiState and draws it with an exhaustive when. This is stateless and easy to preview because it does not create the ViewModel.
collectAsStateWithLifecycle() Subscribes the composable to the ViewModel's StateFlow. When the flow emits a new state, Compose recomposes the UI safely with lifecycle awareness.
onRetry = { viewModel.loadWeather() } Passes a callback that re-runs the load for the current city. The Error UI can retry without knowing how loading works internally.
when (uiState) Chooses Loading, Success, or Error UI. Because WeatherUiState is sealed, missing a state becomes a compiler problem.

When things go wrong

Error handling — one catch, one Error state

A real network call can fail in many ways. The single try/catch in loadWeather() turns any of them into one Error(message) state with a Retry button — never a crash.

What goes wrongThe thrown failureWhat the user sees
No network / airplane modeUnknownHostException“Couldn't load the weather” + the message + Retry
Slow server / timeoutSocketTimeoutExceptionSame Error state; Retry re-runs the load
Malformed / unexpected JSONSerializationExceptionSame Error state (a parse failure, not a crash)
HTTP 4xx / 5xxHttpExceptionSame Error state with the status message
Retry is just loadWeather() again The error screen's button is wired to { viewModel.loadWeather() }. Because loadWeather() resets to Loading and re-enters the same try/catch, recovering from a transient failure costs one tap and zero extra code.
ignoreUnknownKeys handles the friendly case The forecast response is full of fields we don't model (latitude, timezone, current_units, …) — Json { ignoreUnknownKeys = true } decodes successfully and those extra fields are dropped. Only genuinely malformed or type-mismatched JSON throws.

Clear up the likely questions

Common questions, answered directly

These are the questions students usually ask after reading the code. Each answer points back to the exact responsibility in the app.

Why do we need both the DTOs and Weather?

The DTOs belong to the server shape (nested, snake_case); Weather belongs to the app shape (flat, UI-ready). Keeping them separate means the UI and ViewModel do not depend on Open-Meteo's wire format. The mapper is the one controlled place where network data becomes app data.

Why is the JSON modelled as two DTOs instead of one?

The response is nested: a top-level object with a "current" block inside it. The DTOs mirror that nesting — ForecastResponseDto holds a CurrentDto — so decoding lines up key-for-key. The mapper then flattens both levels into a single Weather.

What does @SerialName do here?

The JSON keys are snake_case (temperature_2m) but Kotlin properties are camelCase (temperature). @SerialName("temperature_2m") bridges each one so the decoder knows which JSON key feeds which property. Without it, decoding can't find the value.

Where is the real HTTP request code?

You do not write it by hand. @GET("v1/forecast") plus the @Query params describe the request, and Retrofit.Builder().create(WeatherApi::class.java) generates an implementation of that interface. Calling api.fetchForecast(...) is what triggers the generated HTTP GET.

Why is fetchForecast() marked suspend?

Network I/O takes time. suspend lets the coroutine pause until the response is ready without blocking the main thread. The user can still see a responsive Loading UI while the work is in progress.

Why does fake mode still decode JSON?

The fake repository is offline, but it is not skipping the parsing lesson. It decodes FAKE_FORECAST_JSON into a ForecastResponseDto and flattens that DTO into a Weather, so the same parse-and-flatten idea is exercised without depending on the internet.

What actually causes the screen to change from Loading to Success?

The ViewModel assigns _uiState.value = WeatherUiState.Success(weather). The public uiState emits that new value, collectAsStateWithLifecycle() sees it, and Compose recomposes WeatherContent(), which now takes the Success branch.

Why is there no manual refresh call on the card?

Compose UI is declarative. The screen describes what to draw for each state. When the state value changes, Compose reruns the relevant composables. You update the state, not the card widget.

What happens if the server adds a new field?

Because the JSON parser is configured with ignoreUnknownKeys = true, decoding still succeeds. The unknown field is ignored unless you later add it to a DTO and map it into the domain model.

What happens if the server sends the wrong type?

A wrong type, such as "temperature_2m": "warm" when it must be a number, is not safe to ignore. Decoding throws a serialization exception. The ViewModel catches that exception and publishes Error(message).

Why is WeatherUiState sealed?

A sealed type gives the compiler the complete list of possible states. That makes the when (uiState) in WeatherContent() exhaustive, so adding or missing a state becomes visible during development.

Why do previews pass fake states instead of creating a ViewModel?

Android Studio previews need quick, deterministic UI. A real ViewModel would start coroutines and use a repository. Passing literal states like Success(previewWeather) previews the UI without network, timers, or lifecycle complexity.

Where should new behavior go if the app grows?

New response fields start in the DTOs and toWeather(). New endpoints go in WeatherApi. New data-source choices go behind WeatherRepository. New screen states go in WeatherUiState and must be rendered in WeatherContent().

Guided student exercises

Each exercise has a prediction step, an edit or inspection step, and an answer reveal. The important habit is: predict the state path before running the app.

Exercise 1: switch from real to fake

Predict: What changes if provideWeatherRepository(useFake = true) is used?

Do: Change the flag, run with no network, and watch the same UI states (Loading → Success).

Reveal answer

The ViewModel and UI do not change. Only the repository implementation changes. The fake always publishes Success(weather) from the hardcoded JSON; a live failure becomes Error(message).

Teacher noteAsk students why the ViewModel does not need an if/else for fake vs real.

Exercise 2: tolerate an unmodelled JSON field

Predict: Will parsing fail because the real response also has "timezone" and "current_units"?

Do: Use the DTO mini-lab's Extra field option, then find ignoreUnknownKeys in WeatherApi.kt.

Reveal answer

Parsing succeeds because unknown keys are ignored. Those fields never enter Weather until the app explicitly models and maps them.

Exercise 3: break a required type

Predict: What happens if "temperature_2m" is a string instead of a number?

Do: Use Wrong type in the DTO mini-lab to see the decode failure.

Reveal answer

Decoding throws. The ViewModel catches the exception and publishes Error(message). The app should not crash.

Exercise 4: trace Retry

Predict: Which function runs when the Retry button is tapped?

Do: Find onRetry = { viewModel.loadWeather() }, then follow loadWeather().

Reveal answer

Retry calls loadWeather(). That sets Loading again, tries the repository call again for the current city, then publishes Success or Error.

Exercise 5: add a new UI state on paper

Predict: If you add Empty to WeatherUiState, what else must change?

Do: Do not code it first. List every file and branch that needs a decision.

Reveal answer

At minimum, WeatherContent() needs a new when branch. The ViewModel must decide when to publish Empty, e.g. if a city's reading ever comes back missing.

Exercise 6: explain the whole trip aloud

Predict: Can you say the successful path in fewer than 45 seconds?

Do: Use the seven-step trip section, then try without looking.

Reveal answer

UI calls ViewModel. ViewModel sets Loading. Repository fetches. Retrofit or fake JSON parses into a nested DTO. Mapper flattens it into one Weather. ViewModel publishes Success. Compose redraws the card.

Debug faster

Common mistakes and exactly what they look like

These are the failure patterns students are most likely to hit. Match the symptom to the cause, then inspect the named file.

Missing @Serializable

Symptom: JSON decoding cannot find a serializer for ForecastResponseDto.

Fix: Keep @Serializable on every DTO class and keep the serialization plugin enabled.

Forgot @SerialName

Symptom: A camelCase property like temperature decodes as missing because the JSON key is temperature_2m.

Fix: Add @SerialName("temperature_2m") to bridge the snake_case key.

No INTERNET permission

Symptom: Real mode cannot reach the network even though the device has a connection.

Fix: Keep <uses-permission android:name="android.permission.INTERNET" /> in the manifest.

DTO used directly in UI

Symptom: The nested, snake_cased server shape leaks into screen code, and API changes ripple everywhere.

Fix: Flatten ForecastResponseDto to Weather in the repository boundary.

Error handled in the UI only

Symptom: The app can crash before Compose gets a chance to show a friendly state.

Fix: Keep the try/catch around repository.getWeather() in the ViewModel.

Preview creates a real ViewModel

Symptom: Android Studio preview is slow, flaky, or tries to run app logic.

Fix: Preview WeatherContent() with hand-made WeatherUiState values.

Teacher note Turn each mistake into a five-minute debug drill: show the symptom first, then ask which layer owns the fix.

Try it yourself

Run & inspect

The app builds and tests offline by default. Build from the NetworkParsing/ project directory:

terminal · from NetworkParsing/
./gradlew assembleDebug --console=plain --stacktrace   # build the app
./gradlew testDebugUnitTest                            # mapper + fake-repository unit tests (no network)

Run live (the default) or force offline

  • Live is the default — Open-Meteo is free and keyless, so provideWeatherRepository(useFake = false) hits the real API out of the box.
  • The INTERNET permission is already declared in AndroidManifest.xml, so no manifest edit is needed.
  • Run on a device/emulator with a connection — tap a city chip and it will hit https://api.open-meteo.com/v1/forecast live. To demo offline, pass useFake = true.
