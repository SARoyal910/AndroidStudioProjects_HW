1 What an “API” is, in one minute

An API (Application Programming Interface) is a doorway one program opens so other programs can ask it for things. A web API is that doorway reachable over the internet: you send an HTTP request to a URL, and the server sends back data — almost always as JSON text.

Think of a restaurant: the menu is the API (what you’re allowed to order), your order is the request, and the plate of food is the response. You don’t go into the kitchen — you just use the menu.

Our API: Open-Meteo — a free weather service. We use one endpoint, Forecast, which lives at:

https://api.open-meteo.com/v1/forecast

Open-Meteo is free for non-commercial use and needs no API key, which is why this app works the moment you run it.

2 The request, piece by piece

Every web request is just a URL plus an HTTP method. We only ever do a GET (“fetch me some data”). Here is the exact URL the app builds, color-coded:

https://api.open-meteo.com/v1/forecast?latitude=28.54&longitude=-81.38&current=temperature_2m,relative_humidity_2m,apparent_temperature,weather_code,wind_speed_10m
PieceWhat it isWhy it’s there
https://schemeEncrypted HTTP — always use HTTPS.
api.open-meteo.comhostWhich server to talk to. In Retrofit this is the base URL.
/v1/forecastpathWhich endpoint on that server. In Retrofit this is the @GET("v1/forecast") path.
?…query startA ? begins the list of parameters.
latitude, longitudequery paramsWhere to get the weather. Open-Meteo is coordinate-based (not city names).
currentquery paramWhat readings you want, comma-separated. We ask for temperature, humidity, “feels like”, the weather code, and wind.

Each &name=value pair becomes one @Query("name") parameter in the Kotlin interface — see §6.

4 Read the JSON it sends back

A successful response is a JSON object. It contains a lot we ignore (location, units, timezone) and one block we care about: current. Here is a real, trimmed response:

// GET …/v1/forecast?latitude=28.54&longitude=-81.38&current=…
{
  "latitude": 28.55, "longitude": -81.38, "timezone": "GMT",
  "current_units": { "temperature_2m": "°C", "wind_speed_10m": "km/h" },
  "current": {
    "time": "2026-06-25T12:30",
    "temperature_2m": 26.8,
    "relative_humidity_2m": 41,
    "apparent_temperature": 26.3,
    "weather_code": 0,
    "wind_speed_10m": 8.4
  }
}
Two things to notice. (1) The keys are snake_case (temperature_2m), which doesn’t match Kotlin’s camelCase — we bridge that with @SerialName. (2) weather_code is a number (a WMO code), not text — we decode it ourselves (see the table below).

This is the lookup the app’s wmoDescription(code) function performs.

5 From JSON to a clean object on screen

The app never lets that nested, snake_cased JSON reach the UI. It does this in three steps, in one place each:

JSON key (wire)DTO property (CurrentDto)Domain field (Weather)
current.temperature_2m@SerialName("temperature_2m") temperaturetemperatureC (rounded)
current.apparent_temperature@SerialName("apparent_temperature") feelsLikefeelsLikeC (rounded)
current.relative_humidity_2m@SerialName("relative_humidity_2m") humidityhumidityPercent
current.weather_code@SerialName("weather_code") weatherCodedescription (via wmoDescription)
current.wind_speed_10m@SerialName("wind_speed_10m") windSpeedwindKmh (rounded)

The DTO mirrors the wire exactly; the domain model is flat and UI-ready; the mapper (toWeather()) is the single bridge. If Open-Meteo ever renames a field, only the DTO + mapper change — never the screen.

6 How the app makes the call (the Kotlin)

You declare the endpoint as an interface; Retrofit writes the HTTP code for you. Each @Query becomes one &name=value in the URL from §2.

// WeatherApi.kt
interface WeatherApi {
    @GET("v1/forecast")
    suspend fun fetchForecast(
        @Query("latitude") latitude: Double,
        @Query("longitude") longitude: Double,
        @Query("current") current: String = "temperature_2m,…,wind_speed_10m",
    ): ForecastResponseDto          // JSON object → DTO, parsed for you
}

The repository makes the call, then flattens the DTO into the domain model — and a one-line switch lets it run offline:

// WeatherRepository.kt
class RealWeatherRepository(private val api: WeatherApi = provideWeatherApi()) : WeatherRepository {
    override suspend fun getWeather(city: City): Weather {
        val dto = api.fetchForecast(city.latitude, city.longitude)  // network + parse
        return dto.toWeather(city)                                   // nested DTO → flat domain
    }
}

// REAL by default (free, no key); pass useFake = true to run fully offline.
fun provideWeatherRepository(useFake: Boolean = false): WeatherRepository =
    if (useFake) FakeWeatherRepository() else RealWeatherRepository()

The ViewModel wraps the call in try/catch so a failure becomes a visible Error state instead of a crash — the whole reason the screen has three states. See WeatherViewModel.kt.

7 Coordinates, and adding your own city

Because Open-Meteo wants latitude/longitude, the app ships a small list of cities with their coordinates (PRESET_CITIES in Weather.kt). To add one, find its coordinates and append an entry:

// Weather.kt
val PRESET_CITIES = listOf(
    City("Orlando", "US", 28.5384, -81.3789),
    City("Paris",   "FR", 48.8566,   2.3522),   // ← add a line like this
)
Want a free-text search box instead? Open-Meteo has a separate, also-free Geocoding API that turns a typed name into coordinates: 
https://geocoding-api.open-meteo.com/v1/search?name=Paris&count=1
Call it first to get latitude/longitude, then call the forecast endpoint — a nice “chain two requests in the repository” extension.

8 Run it: live vs offline

The app runs live by default — no setup. On a device or emulator with internet, tap a city chip and you’ll see real current weather.

  • Offline / no network: the request throws and you get the Error card + Retry — by design.
  • Force offline mode (for demos or tests): pass useFake = true to provideWeatherRepository(...); it returns the hardcoded sample after a short delay so you still see Loading → Success.
  • The INTERNET permission is already declared in AndroidManifest.xml — required for the live call.
# Build & run from the project root
./gradlew installDebug
# Or just compile + run the offline unit tests (no internet needed)
./gradlew testDebugUnitTest

10 When an API does need a key

Open-Meteo is unusual in needing no key. Most commercial APIs (OpenWeatherMap, Google Maps, GitHub, Stripe…) give you a secret API key you must send with each request — and which you must never commit to git. The pattern used elsewhere in this repo (see the FirebaseSync / SupabaseSync projects) is:

  1. Put the key in local.properties (already git-ignored) — never in source: 
    # local.properties (NOT committed)
someapi.apiKey=YOUR_SECRET_KEY
  2. Have build.gradle.kts read it into BuildConfig at build time: 
    val props = Properties().apply { rootProject.file("local.properties").inputStream().use { load(it) } }
buildConfigField("String", "SOMEAPI_KEY", "\"${'$'}{props.getProperty(\"someapi.apiKey\") ?: \"\"}\"")
  3. Read BuildConfig.SOMEAPI_KEY in code and send it as a query param or header — falling back to an offline fake when the key is blank.
Security rule of thumb: a key in client code can always be extracted, so treat mobile keys as identifiers, restrict them (by app id / referrer / rate limit) in the provider’s dashboard, and put anything truly sensitive behind your own server. Keys in local.properties stay off GitHub; if one leaks, rotate it.
