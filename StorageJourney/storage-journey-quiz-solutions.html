1. The layers an item travels through Select all that apply

Which of the following are layers a saved item actually travels through on its journey in this app?

Across: MainActivity.kt → StorageJourneyViewModel.kt → ItemRepository.kt → ItemDao.kt → back via the items Flow

  • The Compose UI — the Save button captures the typed textCorrect The UI leg: the OutlinedTextField's value is handed to viewModel.addItem(label). This is hop 1, recorded as JourneyStage.UI.
  • The ViewModel — stamps createdAt, builds the StoredItem, launches a coroutineCorrect In addItem(): StoredItem(label, createdAt = System.currentTimeMillis()) then viewModelScope.launch { repository.save(element) } — hop 2, the ViewModel leg.
  • The Repository — the one storage boundary that hands the element to the DAOCorrect ItemRepository.save() is the single door: it calls dao.insert(item). The ViewModel never touches the DAO directly — hop 3, Repository.
  • Room / SQLite — the @Insert runs a real INSERT and commits the row to journey.dbCorrect Room's generated insert() executes INSERT INTO items(...) and SQLite commits the row to the journey.db file — hop 4, Room · SQLite.
  • A remote REST web service over the networkNot correct There is no networking anywhere in StorageJourney — it is a pure local persistence demo. The data never leaves the device. (Networking is the NetworkParsing / CloudSync projects.)
Takeaway: four layers, all on-device: UI → ViewModel → Repository → Room. The fifth hop (Flow back to the UI) is the return trip, covered in Q4 and Q8.
2. Why insert(item) is suspend Select all that apply

Why is the DAO's insert(item) declared as a suspend function?

Where: ItemDao.kt → @Insert suspend fun insert(item): Long

  • So Room can run the actual disk write off the main thread, keeping the UI smoothCorrect A suspend DAO method lets Room move the blocking disk I/O to a background dispatcher internally, so the main thread is never stalled.
  • So it can be called from a coroutine (viewModelScope.launch) that pauses until SQLite commitsCorrect suspend functions are only callable from a coroutine. The ViewModel launches one; it suspends at dao.insert(...) and resumes once the write is committed — without blocking the thread.
  • So it can return the new row's auto-generated id once the write completesCorrect insert(...): Long returns the freshly assigned primary key. The repository captures it as newId and logs "row #N" in the journey.
  • Because you must always wrap a Room call in Dispatchers.IO yourselfNot correct The opposite is true: because a suspend DAO method already runs the I/O off-thread, wrapping it in withContext(Dispatchers.IO) is redundant. The code comment in ItemDao.kt says exactly this.
  • Because the suspend keyword makes the insert run fasterNot correct suspend is about where work runs (concurrency, not blocking), not raw speed. It doesn't make SQLite write any faster.
Takeaway: suspend = "callable from a coroutine, can pause without blocking." Room uses that to keep disk writes off the UI thread for you — you don't add Dispatchers.IO on top.
3. What KSP does in this project Select all that apply

What does KSP (Kotlin Symbol Processing) do in this project?

Where: build.gradle.kts (the ksp plugin + ksp("androidx.room:room-compiler")), feeding @Entity/@Dao/@Database

  • Generates the concrete ItemDatabase_Impl and ItemDao_Impl classes at build timeCorrect That is why ItemDatabase and itemDao() are abstract — you never write the bodies; KSP emits them into build/generated/ksp/….
  • Reads Room's @Entity, @Dao, and @Database annotations to produce the SQLite codeCorrect KSP is an annotation processor: it inspects those annotations and writes the matching CREATE TABLE / INSERT / SELECT plumbing so you never hand-write SQL boilerplate.
  • Must be applied as a Gradle plugin, or the ksp(room-compiler) dependency does nothingCorrect The com.google.devtools.ksp plugin and the ksp(...) dependency work together: without the plugin, the processor never runs and Room's _Impl classes are never generated — the build fails.
  • Runs the app on the device or emulatorNot correct KSP is a compile-time code generator. Launching the app is done by Gradle's install task + the Android runtime, not by KSP.
  • Encrypts the journey.db file on diskNot correct KSP generates code; it has nothing to do with encryption. The DB is a plain SQLite file (encryption would need SQLCipher or similar, which this project doesn't use).
Takeaway: KSP is the build-time robot that turns your annotated abstract Room declarations into real, runnable SQLite code. It never touches runtime behavior or storage security.
4. The reactive read observeItems(): Flow<List<StoredItem>> Select all that apply

Which statements about the reactive read observeItems(): Flow<List<StoredItem>> are TRUE?

Where: ItemDao.kt → @Query("SELECT * FROM items ORDER BY id DESC") fun observeItems(): Flow<…>

  • Room re-runs the SELECT and emits a new list every time the items table changesCorrect A Room query returning Flow is live: Room watches the table and re-queries on any change, pushing a fresh List down the stream. This is the Flow · UI return trip.
  • It means you never have to write a manual "reload the list" callCorrect The whole point of the reactive read: after a write, the new row arrives on screen automatically — no refresh() button, no re-fetch code.
  • It re-emits on deletes as well as insertsCorrect Room re-emits on any change to the watched table. deleteItem() and clearAll() both trigger a fresh emission — which is why the Journey panel records a Flow leg on delete too.
  • ORDER BY id DESC makes the newest row appear first in the listCorrect Ids auto-increment, so the highest id is the most recently inserted. DESC sorts highest-first, so the item you just saved shows at the top.
  • Like a suspend write, it returns one value once and then stopsNot correct That describes a suspend function (one-shot). A Flow is the opposite — an ongoing stream that keeps emitting as long as someone collects it and the table keeps changing.
Takeaway: suspend writes are one-and-done; a Flow read is a living subscription. That contrast is the heart of "write once, screen updates forever."
5. How the primary key id is assigned Select all that apply

How is each StoredItem's primary key id assigned?

Where: StoredItem.kt (the entity) + ItemDao.insert(): Long

  • The id property defaults to 0 in the data classCorrect val id: Long = 0. Passing 0 is the signal "I don't have an id yet — assign one for me."
  • @PrimaryKey(autoGenerate = true) tells SQLite to assign the next id on insertCorrect With autoGenerate = true, inserting a row whose key is 0 makes SQLite auto-increment and supply the next unique id.
  • dao.insert(...) returns the new id back to the app as a LongCorrect The repository keeps it: val newId = dao.insert(item), then logs "committed the row … as id #$newId." You literally watch SQLite hand the id back.
  • The ViewModel sets id = System.currentTimeMillis() before savingNot correct System.currentTimeMillis() is used for createdAt (the timestamp column), not the id. The ViewModel leaves id at its default 0 so SQLite assigns the real key.
  • You must hand-write a CREATE TABLE statement to define the keyNot correct Room/KSP generate the CREATE TABLE (including the auto-increment key) from the @Entity + @PrimaryKey annotations. No hand-written DDL anywhere.
Takeaway: default id = 0 + autoGenerate = true means "let SQLite number the rows," and insert(): Long reports the number it picked. createdAt is a separate timestamp, not the key.
6. The database singleton ItemDatabase.getInstance Select all that apply

Which statements about the Room database singleton (ItemDatabase.getInstance) are correct?

Where: ItemDatabase.kt → companion object { @Volatile … getInstance(context) }

  • It uses double-checked locking inside synchronized so only one instance is ever builtCorrect INSTANCE ?: synchronized(this) { INSTANCE ?: build… } — the second null-check, inside the lock, stops two racing threads from each building a database.
  • @Volatile makes a write to INSTANCE immediately visible to other threadsCorrect Without @Volatile a thread could see a stale/half-built value. It guarantees the freshly built instance is visible everywhere, which is what makes the double-checked lock correct.
  • It passes applicationContext so it does not leak an ActivityCorrect context.applicationContext is handed to Room.databaseBuilder. The DB outlives any one Activity, so holding an Activity Context would leak it; the app Context lives as long as the process.
  • "journey.db" is the name of the SQLite file it creates on diskCorrect The third argument to Room.databaseBuilder(...) is the on-disk file name — here "journey.db", the file the committed rows live in.
  • Opening two databases over the same file is the recommended approachNot correct Exactly backwards — that is what the singleton prevents. Two database instances over one file is wasteful and can corrupt state, so we build one and cache it.
Takeaway: one process, one database instance, one file. @Volatile + double-checked synchronized guarantees that even under a startup race, and applicationContext keeps it leak-free.
7. What survives — persistence vs. lifecycle Select all that apply

After you save an item, which actions leave the stored rows INTACT (still present afterward)?

Concept: a committed SQLite row lives in journey.db on disk, independent of process & UI lifecycle

  • Rotating the screenCorrect Rotation recreates the Activity but the row never left disk — and the ViewModel survives the config change anyway, so the list comes right back.
  • Killing the app process and relaunching itCorrect In-memory state is gone, but the row was committed to the journey.db file. On relaunch, observeItems() re-reads it and the item reappears. This is the whole point of persistence.
  • Rebooting the deviceCorrect The SQLite file lives in the app's private storage on flash, which survives power cycles. Reboot doesn't touch it.
  • Uninstalling the appNot correct Uninstalling wipes the app's entire private data directory — journey.db and all — so the rows are gone for good.
  • Tapping "Clear storage" (DELETE FROM items)Not correct clearAll() runs DELETE FROM items, which intentionally removes every row. That's a deliberate write, not an accident of lifecycle.
Takeaway: persistence beats lifecycle events (rotate, process death, reboot) because the data is on disk — but it does not beat a deliberate delete or an uninstall that erases the file itself.
8. The items StateFlow in the ViewModel Select all that apply

Which statements about the items StateFlow in the ViewModel are TRUE?

Where: StorageJourneyViewModel.kt → val items = repository.items.onEach{…}.stateIn(…)

  • .stateIn(...) converts Room's cold Flow into a hot StateFlow that always has a valueCorrect stateIn(viewModelScope, …, emptyList()) turns the cold DAO Flow into a hot StateFlow the UI can read synchronously — it always exposes a current .value.
  • SharingStarted.WhileSubscribed(5000) keeps it hot for 5s after the last collector leavesCorrect The 5-second grace window means a quick rotation (collector briefly detaches, then re-attaches) doesn't tear down and rebuild the upstream query.
  • It is given an empty list as its initial valueCorrect The third stateIn argument is emptyList() — what the screen renders for the first frame before Room's first emission arrives.
  • .onEach { } records the FLOW leg each time Room re-emits a new listCorrect The .onEach { list -> journeyLog.record(JourneyStage.FLOW, …) } side-effect fires on every emission — that's how the Flow · UI hop appears in the panel.
  • Collecting it blocks the main thread until the database query finishesNot correct Collecting a Flow suspends — it never blocks. The query runs off-thread and emissions are delivered back without freezing the UI; that's the entire reason for the coroutine/Flow design.
Takeaway: stateIn bridges Room's cold stream to an always-has-a-value hot StateFlow the UI can collect lifecycle-safely — initialized empty, kept warm for 5s, and never blocking.
9. Why AndroidViewModel + a ViewModelProvider.Factory Select all that apply

Why does StorageJourneyViewModel extend AndroidViewModel and use a ViewModelProvider.Factory?

Where: StorageJourneyViewModel(app: Application) : AndroidViewModel(app) + the companion object { factory(app) }

  • AndroidViewModel supplies an Application Context to build Room without leaking an ActivityCorrect It needs a Context for ItemDatabase.getInstance(app). The Application Context is safe to hold for the ViewModel's lifetime; an Activity Context would leak.
  • The Factory is needed because AndroidViewModel requires the Application be passed inCorrect AndroidViewModel has no zero-arg constructor — it takes an Application. The default ViewModel factory can't supply that, so the tiny factory(app) constructs it explicitly.
  • The ViewModel survives configuration changes such as rotationCorrect That's the core ViewModel guarantee: it outlives Activity recreation, so the screen's state (and the in-flight save) survives a rotation.
  • viewModelScope is a CoroutineScope tied to the ViewModel's lifetime, used for the suspend writesCorrect viewModelScope.launch { repository.save(...) } runs the suspend write in a scope that auto-cancels when the ViewModel is cleared — no leaked coroutines.
  • The Factory pulls in a dependency-injection library such as HiltNot correct The opposite: the hand-written factory exists so the project needs no DI library. The code comment calls it "a tiny factory rather than pulling in a dependency-injection library."
Takeaway: AndroidViewModel gives a leak-safe Application Context; the manual Factory feeds that Context in without any DI framework; and the ViewModel + viewModelScope keep state and coroutines alive exactly as long as the screen needs them.
10. The JourneyLog narrator Select all that apply

Which statements about the JourneyLog narrator are TRUE?

Where: journey/JourneyLog.kt (and its JVM test JourneyLogTest.kt)

  • It holds steps in a MutableStateFlow and exposes a read-only StateFlow via asStateFlow()Correct Private _steps = MutableStateFlow(emptyList()); public val steps = _steps.asStateFlow(). Only the log can mutate it; the UI can only read.
  • It has no Android dependencies, which lets it be unit-tested on the JVMCorrect No android.* imports — the clock is injected as now: () -> Long. That's exactly why JourneyLogTest.kt runs on a plain JVM with a fake clock, no emulator.
  • update { old -> old + step } is an atomic read-modify-write that builds a new listCorrect StateFlow.update reads the current list, appends the step into a new list, and stores it atomically (retrying on contention). The new-list identity is what makes StateFlow re-emit.
  • It uses the same reactive Flow → Compose pattern Room uses for the dataCorrect The narration rides the identical mechanism as the data: a StateFlow the UI collects, recomposing the panel on each recorded step — the journey trace is reactive state.
  • It persists the journey steps by writing them into the SQLite items tableNot correct The steps are purely in-memory UI state — they vanish on reset() and process death. Only StoredItem rows go into the items table; the narrator never touches the database.
Takeaway: JourneyLog is a tiny, Android-free, testable StateFlow recorder. It describes the trip using the same reactive pattern Room uses for the data — but it stores nothing on disk.

One paragraph that answers all ten

A tap in the UI calls addItem(); the ViewModel stamps createdAt, builds a StoredItem (id left 0), and viewModelScope.launches a coroutine; the Repository — the one storage door — calls the suspend dao.insert(); Room (its _Impl classes generated by KSP from the annotations) runs a real INSERT into journey.db, lets SQLite auto-assign the id, and returns it as a Long; because observeItems() returns a Flow, Room re-runs SELECT … ORDER BY id DESC and pushes a new list, which stateIn exposes as a hot StateFlow the screen collects — recomposing with the new row on top, no manual refresh. The row now survives rotation, process death, and reboot (but not uninstall or "Clear storage"). The whole trip is narrated by JourneyLog, an Android-free StateFlow recorder that uses the very same reactive pattern as the data it describes.
