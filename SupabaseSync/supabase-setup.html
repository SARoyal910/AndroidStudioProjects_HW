Supabase setup & API guide

Connect SupabaseSync to a real Supabase Postgres.

By default the app runs fully offline against an in-memory FakeCloudApi, so it builds and tests with no account. This guide takes you the rest of the way: create a Supabase project, create the notes table with Row-Level Security, copy two credentials, flip one flag, and watch a note you add on the phone appear as a row in your Supabase table.

The shape of the whole thing

The cloud here is a hosted Postgres database with an auto-generated REST API (PostgREST). Your data lives as rows in the notes table. The app talks to it through exactly two calls: from("notes").upsert(...) to push and from("notes").select { … } to pull. Everything else is the same offline-first machinery (Room is the single source of truth) covered in How a note syncs.

Two badges you will see throughout this guide
Supabase ➕ Adding to Supabase · inserts a row into notes What actually happens in your Supabase project — Adding / Updating / Deleting / Reading from the notes table, or (muted) nothing leaves the device.
API call 📞 Supabase API · from("notes").upsert(note) The exact supabase-kt SDK call that fires — the literal Kotlin you would find in data/CloudApi.kt.

The cloud

Supabase = hosted Postgres + PostgREST. No built-in offline cache — which is why we pair it with Room.

Your data

Five columns in one table: id, title, body, updated_at, deleted.

The gate

The public anon key identifies the app; RLS policies are what actually allow or deny each row operation.

Before you start

Prerequisites

You can do everything here from a browser; the CLI route is optional.

  • A free Supabase account (supabase.com/dashboard). One free project is plenty.
  • This project opening and building in Android Studio (it already builds offline against the Fake).
  • A device/emulator with internet — the INTERNET permission is already declared in the manifest.
  • Optional: the Supabase CLI (npm i -g supabase or brew install supabase/tap/supabase) if you prefer creating the table from your terminal instead of the dashboard.
  • Optional: curl (pre-installed on macOS/Linux) for the REST sanity check in section 11.
No setup? Still works.

If you skip this entire guide, the app still runs — provideCloudApi(useFake = true) hands back the in-memory fake cloud. This guide is only needed to go live.

Set up the backend

Create the project, the notes table, and its RLS policies

There are two routes to the same result. Pick one. The dashboard route is all clicks + one SQL paste; the CLI / Management route is reproducible from a terminal. Either way you end up with: a project, a notes table whose columns match NoteDto, and RLS policies that let the anon key read and write it.

Route A — the dashboard

  1. Go to supabase.com/dashboard and click New project. Give it a name, a database password (you will not need it for this app), and a region close to you. Wait ~1 minute for it to provision.
  2. Open the SQL Editor (left rail) and paste the SQL block below, then Run. It creates the table and the four RLS policies in one shot.
  3. Confirm in the Table Editor that a notes table now exists with five columns.
Why these exact column names?

The column names are the contract with the app. Four of them (id, title, body, deleted) spell the same in Kotlin and Postgres. The timestamp is the one mismatch: Postgres uses snake_case updated_at, the Kotlin property is updatedAt. The @SerialName("updated_at") annotation in data/Note.kt bridges that one boundary — so the column must be named updated_at for the gt("updated_at", since) filter and JSON decode to line up end-to-end.

Supabase SQL Editor — create table + RLS (verbatim from the README)
create table public.notes (
  id         text primary key,
  title      text not null,
  body       text not null default '',
  updated_at bigint not null,
  deleted    boolean not null default false
);

-- For a quick teaching demo, allow the public anon key to read/write this one table.
-- (In production you would scope these policies to authenticated users.)
alter table public.notes enable row level security;
create policy "demo anon read"  on public.notes for select using (true);
create policy "demo anon write" on public.notes for insert with check (true);
create policy "demo anon update" on public.notes for update using (true) with check (true);
create policy "demo anon delete" on public.notes for delete using (true);

Note updated_at is a bigint — the app stores epoch milliseconds (a Kotlin Long), not a Postgres timestamptz. That keeps last-write-wins a single integer comparison on both sides.

Route B — the Supabase CLI / Management API

Prefer your terminal? The same SQL can be applied through the CLI as a migration, or pushed straight to a project. First log in and link (or create) a project:

terminal — supabase CLI
# 1) authenticate the CLI against your Supabase account (opens a browser)
supabase login

# 2) link this folder to your project (grab the ref from the dashboard URL or `supabase projects list`)
supabase link --project-ref <your-project-ref>

# 3) start a migration that holds the create-table + RLS SQL above
supabase migration new create_notes_table
#    -> writes supabase/migrations/<timestamp>_create_notes_table.sql
#    paste the SQL block from Route A into that file, then push it:
supabase db push

If you do not want a local migration folder, you can run the SQL directly against the linked database, or hit the Management API query endpoint with your personal access token (find tokens under Account ▸ Access Tokens):

terminal — run SQL directly
# Option 1: pipe the SQL file straight at the linked Postgres
supabase db push   # (after `migration new`, as above)

# Option 2: Management API — POST the SQL to the query endpoint
curl -X POST "https://api.supabase.com/v1/projects/<ref>/database/query" \
  -H "Authorization: Bearer $SUPABASE_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"query":"create table public.notes ( id text primary key, title text not null, body text not null default '"'"''"'"', updated_at bigint not null, deleted boolean not null default false );"}'
Don't forget the policies

Creating the table is not enough. With RLS enabled but no policies, every request is denied (a 401/permission error). You must also run the four create policy … statements (or the app will see "permission denied for table notes"). See Troubleshooting.

Set up the backend

The two credentials: Project URL + anon key

supabase-kt needs exactly two strings. Both live in the dashboard under Project Settings ▸ Data API (older UIs label it just "API"):

ValueWhereWhat it is / does
SUPABASE_URL Project Settings ▸ Data API ▸ Project URL Looks like https://<ref>.supabase.co. Every HTTP call the client makes goes to this host.
SUPABASE_ANON_KEY Project Settings ▸ Data API ▸ Project API keys ▸ anon / public A long JWT. It identifies the app as the anonymous role; it does not grant access on its own — RLS does. It is safe to ship in client code.

Paste both into the constants at the top of data/CloudApi.kt (the sample values already there point at a throwaway demo project — replace them with yours):

data/CloudApi.kt — paste your two values here
private const val SUPABASE_URL = "https://<your-ref>.supabase.co"
private const val SUPABASE_ANON_KEY = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...your-anon-key..."
Why shipping the anon key is fine

Unlike a service-role key, the anon key is publishable. The only thing it can do is what your RLS policies permit for the anonymous role. The demo policies above are wide open so the app works immediately; tighten them (e.g. scope to authenticated users) before storing anything real.

Wire up the app

The Gradle dependencies

These are already in the project — this section explains what each line buys you so nothing is magic. The cloud transport is three coordinated pieces: the supabase-kt client, a Ktor HTTP engine it routes through, and kotlinx.serialization for the JSON DTOs.

build.gradle.kts — plugins
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.compose)
    alias(libs.plugins.ksp)                 // Room's annotation processor
    // kotlinx.serialization compiler plugin — lets @Serializable NoteDto encode to/from the JSON
    // that supabase-kt sends to the Supabase (PostgREST) API.
    alias(libs.plugins.kotlin.serialization)
}
build.gradle.kts — the cloud transport dependencies
// Cloud transport: Supabase (hosted Postgres) reached over Ktor + kotlinx.serialization.
implementation(platform(libs.supabase.bom))         // one aligned set of supabase-* versions
implementation(libs.supabase.postgrest)             // the Postgrest (table CRUD) module
implementation(libs.ktor.client.okhttp)             // the HTTP engine supabase-kt routes through
implementation(libs.kotlinx.serialization.json)     // (de)serialize NoteDto to/from JSON
gradle/libs.versions.toml — the versions + coordinates
# supabase-kt = the Kotlin client for Supabase (our REAL cloud: a hosted Postgres database).
# Pinned to the 3.2.x line because it is built against Kotlin 2.2 (matching `kotlin`).
supabaseBom = "3.2.2"
# Ktor is the HTTP engine supabase-kt talks through. 3.2.2 is the version supabase-kt 3.2.2 uses.
ktor = "3.2.2"
# kotlinx.serialization = JSON encode/decode for the cloud DTOs (@Serializable NoteDto).
kotlinxSerialization = "1.9.0"

# supabase-bom is a "platform" dependency; the supabase-* modules take their version from it.
supabase-bom = { group = "io.github.jan-tennert.supabase", name = "bom", version.ref = "supabaseBom" }
supabase-postgrest = { group = "io.github.jan-tennert.supabase", name = "postgrest-kt" }
# A Ktor client ENGINE supabase-kt routes its HTTP through (OkHttp under the hood).
ktor-client-okhttp = { group = "io.ktor", name = "ktor-client-okhttp", version.ref = "ktor" }
kotlinx-serialization-json = { group = "org.jetbrains.kotlinx", name = "kotlinx-serialization-json", version.ref = "kotlinxSerialization" }
The BoM

platform(libs.supabase.bom) is a bill of materials: a "platform" dependency that pins one aligned set of versions for every supabase-* module. That is why supabase.postgrest below it needs no version — it inherits 3.2.2 from the BoM, so the modules can never drift out of sync.

Why a Ktor engine?

supabase-kt does not bundle its own HTTP engine — you supply one. ktor-client-okhttp provides OkHttp under the hood. The engine is discovered from the classpath automatically; you never wire it up in code. Omit it and the client throws at runtime with "no engine found" (see Troubleshooting).

serialization

The compiler plugin generates encode/decode logic for @Serializable classes at build time; the kotlinx-serialization-json runtime is what supabase-kt's decodeList/decodeSingle use to turn PostgREST's JSON into NoteDto.

Wire up the app

Initialise the client — line by line

The entire connection is one expression in SupabaseCloudApi: build a single shared client and install the Postgrest plugin onto it.

data/CloudApi.kt — SupabaseCloudApi (the init)
private val client = createSupabaseClient(supabaseUrl, supabaseKey) {
    install(Postgrest)                                  // enable table CRUD (the PostgREST API)
}
API call 📞 Supabase API · createSupabaseClient(url, key) { … } API call 📞 Supabase API · install(Postgrest)
  1. createSupabaseClient(supabaseUrl, supabaseKey) { … } — the factory that builds the one shared client. supabaseUrl is your Project URL (every HTTP call goes there); supabaseKey is the anon key (identifies the app — RLS is the real gate). The trailing lambda is the config block.
  2. install(Postgrest) — registers the Postgrest plugin, the module that exposes from("tableName").select{}, .upsert(), etc. Without this line, a call like client.from("notes") throws at runtime.
  3. The HTTP transport is implicit: supabase-kt finds the ktor-client-okhttp engine on the classpath and sends requests through it. No engine code appears here.
No network yet

Building the client and installing the plugin are local — no request is sent. The first actual round trip happens when sync() calls from("notes").select{…} or .upsert(…). That is exactly why these two pills are API-call pills with no cloud-effect pill: they touch no rows.

Wire up the app

Flip the switch: Fake → real Supabase

One factory decides which cloud the app uses. The useFake flag flows from the ViewModel → the repository → provideCloudApi.

data/CloudApi.kt — provideCloudApi (the factory)
private val fakeCloud: FakeCloudApi by lazy { FakeCloudApi() }

fun provideCloudApi(useFake: Boolean = true): CloudApi =
    if (useFake) fakeCloud else SupabaseCloudApi()

In this project the lever is already pulled to the real cloud in the ViewModel:

SupabaseSyncViewModel.kt
// useFakeCloud = false → talk to the REAL Supabase Postgres configured in data/CloudApi.kt.
// Flip back to NoteRepository.get(app) (default true) to demo fully offline against the Fake.
private val repo = NoteRepository.get(app, useFakeCloud = false)

The flag is forwarded through the repository's singleton getter into the factory:

NoteRepository.kt — get(...) passes the flag to provideCloudApi
fun get(context: Context, useFakeCloud: Boolean = true): NoteRepository =
    INSTANCE ?: synchronized(this) {
        INSTANCE ?: NoteRepository(
            dao = SupabaseSyncDatabase.getInstance(context).noteDao(),
            cloud = provideCloudApi(useFakeCloud),       // ← the switch
        ).also { INSTANCE = it }
    }
It is a process singleton

NoteRepository.get(...) caches the first instance it builds. If you change the flag, do a clean reinstall (or kill the process) so the new cloud is actually constructed — the cached instance will otherwise keep the old one. The SyncWorker calls get(applicationContext) with the default, so it reuses whatever the UI built.

The real code

Every Supabase API this app uses

The whole Supabase surface this app touches fits in the two methods of SupabaseCloudApi plus the init. The frame to keep in your head:

The two operations push() = from("notes").upsert(note){select()}.decodeSingle().  ·  pull() = from("notes").select { filter { gt } ; order }.decodeList().

Below is one block per API: the real snippet from data/CloudApi.kt (or Note.kt), a plain explanation, an API-call pill, and — where the call changes cloud data — a cloud-effect pill.

    install(Postgrest)
}

Builds the one shared SupabaseClient from your Project URL + anon key. No network happens here — it is just a configured client object. The trailing lambda is where you install plugins.

API call 📞 Supabase API · createSupabaseClient(url, key){ … }

Called when you… first construct SupabaseCloudApi() (i.e. when the repository is built with useFakeCloud = false). Once, not per request.

API call 📞 Supabase API · client.from("notes")

Called when you… add, edit, delete, or sync — every push() and pullSince() starts from this reference.

.select { filter { gt("updated_at", since) }; order(…) } → query pullSince()
data/CloudApi.kt — pullSince()
client.from("notes").select {
    filter { gt("updated_at", since) }              // WHERE updated_at > :since
    order("updated_at", Order.ASCENDING)            // ORDER BY updated_at ASC (oldest-first)
}

Adds a SELECT, then narrows it. filter { gt("updated_at", since) } appends WHERE updated_at > :since — note the snake_case column name, not the Kotlin property. order("updated_at", Order.ASCENDING) sorts oldest-first so the pull cursor stays correct even if the network drops mid-batch.

Called when you… sync/pull (trace 4). Never on add/edit/delete.

.decodeList<NoteDto>() → List<NoteDto> pullSince()
data/CloudApi.kt — pullSince()
    …
    }.decodeList<NoteDto>()   // fire the GET, deserialize the JSON array into List<NoteDto>

This is the moment the read query actually executes (fires the HTTP GET). It receives PostgREST's JSON array and deserialises it into a List<NoteDto> via kotlinx.serialization — which works because NoteDto is @Serializable with the @SerialName("updated_at") mapping.

Supabase ⬇️ Reading from Supabase · selects matching rows (this is when the GET fires) API call 📞 Supabase API · .decodeList<NoteDto>()

Called when you… sync/pull — the execution step of pullSince().

.upsert(note) { select() } → query push()
data/CloudApi.kt — push()
client.from("notes").upsert(note) { select() }

Sends the NoteDto as JSON and runs INSERT INTO notes … ON CONFLICT (id) DO UPDATE — so it creates a new row OR updates the existing one with the same id. The { select() } option appends Prefer: return=representation, telling Postgres to return the stored row in the response body (without it the body would be empty and there would be nothing to decode).

Supabase ➕ Adding / ✏️ Updating Supabase · upsert = insert or update; returns the stored row API call 📞 Supabase API · from("notes").upsert(note){select()}

Called when you… add (creates), edit (updates), or delete (upserts with deleted=true). All three push through this one call — traces 1, 2, 3.

.decodeSingle<NoteDto>() → NoteDto push()
data/CloudApi.kt — push()
client.from("notes").upsert(note) { select() }.decodeSingle<NoteDto>()

Executes the upsert (fires the write) and deserialises the single returned JSON row into one NoteDto — the server's authoritative copy. This is the value the repository re-saves into Room as SYNCED.

@Serializable + @SerialName("updated_at") (Note.kt) the DTO contract
data/Note.kt
@Serializable
data class NoteDto(
    val id: String,
    val title: String,
    val body: String,
    @SerialName("updated_at") val updatedAt: Long,  // bridges snake_case Postgres → camelCase Kotlin
    val deleted: Boolean = false,
)

@Serializable makes the compiler generate JSON encode/decode for the DTO — without it decodeList/decodeSingle could not work. @SerialName("updated_at") is the one place the snake_case Postgres column maps to the camelCase Kotlin property; the Postgres column must be named updated_at for it to line up.

API call 📞 Supabase API · @Serializable / @SerialName("updated_at") (DTO mapping)

Called when you… push or pull — the (de)serialisation runs on every upsert/select round trip. No cloud-effect pill: this is shaping, not a row change.

The Ktor engine (ktor-client-okhttp) transport
build.gradle.kts (no code call — classpath only)
implementation(libs.ktor.client.okhttp)   // discovered from the classpath automatically

supabase-kt does not bundle an HTTP engine; it picks one off the classpath. This project supplies OkHttp via Ktor. There is no Kotlin call to write — but every upsert/select physically travels through this engine. Remove the dependency and the client throws "no engine" at the first request.

Run it, and verify the round trip

With the switch flipped (section 7) and your two credentials in place (section 4), build and install on a networked device:

terminal
JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" \
  ./gradlew :compileDebugKotlin     # type-check + Room KSP codegen
./gradlew installDebug              # build & install on a running device/emulator

What to watch in the app

  • Add a note. Its badge starts ⏳ Pending, then flips to ✓ Synced once the push succeeds.
  • Open the Supabase Table Editor — the same row appears (and on a second device after it syncs).
  • Toggle airplane mode: writes queue as Pending (the banner turns amber), then drain to Synced when you reconnect.
  • Watch Logcat for OkHttp/Ktor request lines and any RLS/permission errors if a write is rejected.

A REST sanity check (no app needed)

You can hit the same PostgREST endpoint the app uses, straight from a terminal, to confirm the table and policies are live. Use your Project URL + anon key:

terminal — curl the REST API
# read rows (mirrors the app's pull). The anon key goes in BOTH headers.
curl "https://<your-ref>.supabase.co/rest/v1/notes?select=*" \
  -H "apikey: $SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_ANON_KEY"

# insert a row (mirrors the app's push / upsert)
curl -X POST "https://<your-ref>.supabase.co/rest/v1/notes" \
  -H "apikey: $SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_ANON_KEY" \
  -H "Content-Type: application/json" \
  -H "Prefer: return=representation" \
  -d '{"id":"curl-test","title":"hi","body":"from curl","updated_at":1,"deleted":false}'
Reading the result

A JSON array (even []) means the URL, key, table, and RLS read policy are all good. A 401/permission message means the key or an RLS policy is the problem; relation "notes" does not exist means the table was never created. The Prefer: return=representation header is the curl equivalent of the app's { select() }.

Troubleshooting

SymptomLikely causeFix
401 / "permission denied for table notes" / "new row violates row-level security policy" RLS is enabled but the needed policy is missing (this is the #1 issue). Enabling RLS with no policy denies everything. Run all four create policy … statements from section 3 (read and write/update/delete). Confirm under Authentication ▸ Policies.
Connection refused / DNS errors / 404 on every call Wrong Project URL or anon key, or you used the service-role key. Re-copy both from Project Settings ▸ Data API into the constants in data/CloudApi.kt. The URL is https://<ref>.supabase.co; the key is the anon / public one.
relation "public.notes" does not exist / 404 for /rest/v1/notes The table was never created (or created in a different schema). Run the create-table SQL from section 3 in the SQL Editor (or via the CLI migration). Verify it shows in the Table Editor under public.
Pulls return rows but updatedAt is always 0 / decode fails on the timestamp / writes never match on edit Column-name mismatch: the column is not named updated_at, or the @SerialName("updated_at") mapping was changed. The Postgres column MUST be updated_at (snake_case) and the DTO MUST keep @SerialName("updated_at") on updatedAt. The gt("updated_at", since) filter uses the column name too.
IllegalStateException "no engine" / "Failed to find HttpClientEngine" at the first request The Ktor engine is missing from the classpath. Keep implementation(libs.ktor.client.okhttp) in build.gradle.kts. supabase-kt does not bundle an engine; it discovers OkHttp from the classpath.
Notes stay ⏳ Pending forever / show ⚠ Failed Every push is throwing (offline, RLS denial, or wrong creds), so sync() bubbles up and the SyncWorker keeps retrying. Fix the underlying cause above (RLS, URL/key, engine). The local data is safe in Room the whole time; once a push succeeds the row flips to Synced.
Switching useFakeCloud seems to have no effect NoteRepository.get(...) is a process singleton — the first cloud built is cached. Do a clean reinstall (or kill the process) after changing the flag so the new SupabaseCloudApi is constructed.
Still stuck?

Run the curl read from section 11 first — it isolates whether the problem is in your Supabase project (URL/key/table/RLS) or in the app. If curl works but the app does not, the issue is on the app side (the switch, the engine dependency, or the @SerialName mapping).
