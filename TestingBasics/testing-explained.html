How to write tests — an interactive intro

A test is just code that checks your other code. This page walks through the TestingBasics app — a tiny tip calculator — and shows exactly how its tests are written, named, and run. There's a live test runner below: edit the calculator, press Run tests, and watch tests pass or fail in real time. The same ideas apply to Homework 7 and every project in this repo.

1. What a test is, and why bother

You already test your code — by running the app and eyeballing it. A unit test just writes that check down so the computer can repeat it instantly, forever.

A unit test calls one piece of your code with a known input and asserts the output is what you expect. If the output is wrong, the test fails and tells you so. Why it's worth it:

  • It's fast. The TipCalculator tests run on your laptop in milliseconds — no emulator.
  • It catches mistakes. Change the math by accident and a red test tells you immediately.
  • It's a safety net. You can refactor fearlessly: if the tests stay green, you didn't break anything.
  • It documents. A well-named test like tipAmount of 100 at 20 percent is 20 says what the code is supposed to do.
The golden rule: the easiest code to test is pure code — functions that take inputs and return outputs with no UI, no clock, no network, no randomness. That's why TipCalculator.kt has zero Android imports. Keep your logic pure and testing is easy; bury it inside a button's onClick and it's hard.

2. The anatomy of a test — Arrange, Act, Assert

Almost every test has the same three beats. Once you see them, you can read (and write) any test.

Arrange
val bill = 100.0
val tipPercent = 20 // set up the inputs
Act
val tip = TipCalculator.tipAmount(bill, tipPercent) // call the real code
Assert
assertEquals(20.0, tip, 0.001) // expected, actual, tolerance

Put together, that's one complete test. In Kotlin/JUnit it looks like this:

import org.junit.Test
import org.junit.Assert.assertEquals

class TipCalculatorTest {

    // The @Test annotation marks this function as a test to run.
    // The backtick name is a readable sentence describing the behaviour.
    @Test
    fun `tipAmount of 100 at 20 percent is 20`() {
        val bill = 100.0                                  // Arrange
        val tip = TipCalculator.tipAmount(bill, 20)        // Act
        assertEquals(20.0, tip, 0.001)                     // Assert
    }
}
Why the 0.001? tip is a Double, and computers can't store most decimals exactly (famously, 0.1 + 0.2 == 0.30000000000000004). So when comparing decimals you allow a tiny tolerance — the delta. 0.001 means "within a tenth of a cent." For exact (non-floating-point) types — Int, String, Boolean — you don't need a delta; those are compared exactly.

    4. How to add a test in Android Studio

    In a real project the steps are short — Android Studio scaffolds the folders for you.

    1

    Find the test folder

    Every project has src/test/java/<your.package>/ for fast JVM unit tests, and src/androidTest/java/... for on-device UI tests. Put logic tests in test.

    2

    Make a test class

    Create a Kotlin file like TipCalculatorTest.kt with a class inside. (Shortcut: open the class you want to test, press ⌘⇧T ▸ Create New Test…, and Studio generates the file in the right place.)

    3

    Write a function annotated @Test

    Arrange inputs, Act by calling your code, Assert the result with assertEquals (or a friend from the table below). Name it after the behaviour it checks.

    4

    Run it — watch for green

    Click the green ▶ in the gutter next to the class (runs all) or next to one @Test. A green ✓ = pass; a red ✗ shows the failing line and the expected-vs-actual. From a terminal: ./gradlew testDebugUnitTest.

    5. The assertions you'll actually use

    An assertion is the line that decides pass or fail. JUnit gives you a handful; these cover almost everything.

    AssertionUse it when…
    assertEquals(expected, actual)two values should match (Int, String, Boolean…).
    assertEquals(expected, actual, delta)comparing Double/Float — allow a small tolerance.
    assertTrue(condition) / assertFalse(...)the result is a yes/no fact ("is it positive?").
    assertNull(x) / assertNotNull(x)checking whether something is null.
    assertThrows(Type::class.java) { ... }bad input should throw — proves your guards fire.

    Every assertion takes an optional first message argument shown when it fails, e.g. assertTrue("tip should be ≥ 0 but was $tip", tip >= 0.0) — worth adding so a red test reads clearly.

    6. Habits that make tests pay off

    • One behaviour per test. If "10%" breaks but "15%" passes, separate tests tell you exactly which.
    • Name the behaviour, not the method. perPerson splits the total evenly beats test1. When it fails, the name alone tells you what's wrong.
    • Test the edges. Zero, one, empty, the maximum, the "impossible" input. Bugs live at the boundaries — that's why we test bill = 0 and people = 0.
    • Red, green, refactor. Write a test (it fails — red), make it pass (green), then clean up the code with the test guarding you. Some people even write the test first.
    • Keep logic pure. The more your code is plain input→output functions, the more of it you can test this easily. Hard-to-test code is usually a hint to pull the logic out of the UI.
