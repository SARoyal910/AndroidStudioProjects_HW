Homework 8 — PokéDex: a live-API app that ties the whole course together

This is the capstone. You'll build a complete Android app that fetches real, live data from a free public API — the PokéAPI, which needs no API key and no account — then combines every skill from the course around it: a Compose UI, Navigation 3 for a list → detail flow, a ViewModel exposing Loading / Success / Error state, Retrofit + kotlinx.serialization for the network calls, and a Room cache so the app still works offline — all proven by unit tests. You build it from a fresh project; this assignment is the spec.

Project: PokeDex (you create it) Live API: PokéAPI — free, no key, no account Tools: Android Studio + emulator/device + internet
▶ Projects, Explained ← Homework 7

Up to now each assignment isolated one idea — a navigation pattern, some Compose layout, a pure game rules layer, a storage API. A real app is all of them at once. In this capstone you assemble the same pieces you've already practised into one coherent app, and you add the one big skill the course saves for last: talking to a server over the network. The data is real and live — actual Pokémon served by PokéAPI, a large, free, keyless public API used in thousands of tutorials.

The app — "PokéDex". A list screen shows the first 151 Pokémon (name + Pokédex number), fetched live from the API. Tap any row to navigate to a detail screen that fetches that one Pokémon and shows its number, its types (as emoji), its height (m) and weight (kg). Every fetched result is cached in a Room database, so when you reopen the app — or open it with the network off — the list and any Pokémon you've already viewed are still there. Each screen explicitly renders Loading, Success, and Error.

Why this API (and the rule it must obey)

The course's hard rule for network homework: the API must be free, need no API key, and need no account. PokéAPI satisfies all three — you just send a plain HTTPS GET and parse the JSON. (It's the same kind of open, keyless endpoint you already met in NetworkParsing, which used Open-Meteo.) PokéAPI asks every app to cache responses locally rather than re-fetch — which is exactly why the Room cache below isn't just bonus polish, it's part of using the API correctly.

You are not learning six new things. You're learning one new thing (network calls) and wiring it to five you already know. Each layer below has a reference project in this repo you can open side-by-side:

The API you'll call read this first

Two endpoints, both plain GET, both keyless. Model your DTOs from the JSON shown here — this is your "wire contract," exactly like the Open-Meteo sample in NetworkParsing.

Endpoint 1 — the list

GET https://pokeapi.co/api/v2/pokemon?limit=151&offset=0

Returns one object with a results array. Each entry has only a name and a url — the Pokédex number is the trailing number in that url (.../pokemon/1/). Trimmed sample:

{
  "count": 1302,
  "next": "https://pokeapi.co/api/v2/pokemon?offset=151&limit=151",
  "results": [
    { "name": "bulbasaur",  "url": "https://pokeapi.co/api/v2/pokemon/1/" },
    { "name": "ivysaur",    "url": "https://pokeapi.co/api/v2/pokemon/2/" },
    { "name": "venusaur",   "url": "https://pokeapi.co/api/v2/pokemon/3/" }
    // … 148 more
  ]
}
Endpoint 2 — one Pokémon's detail

GET https://pokeapi.co/api/v2/pokemon/{name}  (e.g. .../pokemon/bulbasaur)

Returns a large object; you only model the fields you show. Note the units: height is in decimetres (divide by 10 → metres) and weight is in hectograms (divide by 10 → kg). types is a nested array. Trimmed sample:

{
  "id": 1,
  "name": "bulbasaur",
  "height": 7,            // decimetres -> 0.7 m
  "weight": 69,           // hectograms -> 6.9 kg
  "types": [
    { "slot": 1, "type": { "name": "grass"  } },
    { "slot": 2, "type": { "name": "poison" } }
  ]
  // … sprites, stats, abilities, moves, … (ignored)
}
Robustness: the real response has dozens of fields you don't model. Configure the JSON parser with Json { ignoreUnknownKeys = true } — exactly as NetworkParsing does — or decoding will throw on the first unmodelled key. This is the single most important network-robustness setting.

Part B. The network layer — DTOs, API, domain, mapper Android Studio

This is the one genuinely new layer. It's the same four-piece shape as NetworkParsing's Weather.kt + WeatherApi.kt: DTOs (the wire shape) → a Retrofit API interface → a clean domain model → a mapper that converts and decodes the server's values. Open NetworkParsing beside you and follow it field-for-field.

Model the wire, the domain, and the bridge between them
B1DTOs — the exact JSON shapePokemon.kt

Write @Serializable data classes mirroring the two JSON samples above: a list response holding a list of summaries (name, url), and a detail object (id, height, weight, and a nested types list whose entries each wrap a type { name }). Use @SerialName only where a Kotlin name must differ from the JSON key. DTOs carry the network's shape and nothing else.

Think about: how many classes does the nested types array force you to declare? Which fields can you simply leave out of the DTO (and what setting lets you)?

B2Retrofit API — two suspend GETsPokemonApi.kt

Declare an interface PokemonApi with two suspend functions: one for the list (@GET("pokemon") with @Query params limit and offset) and one for the detail (@GET("pokemon/{name}") with an @Path for the name). Then a providePokemonApi() that builds Retrofit with baseUrl("https://pokeapi.co/api/v2/") and the kotlinx.serialization converter. This is a near-verbatim copy of provideWeatherApi().

Two annotations, two jobs: @Query appends ?limit=151 to the URL; @Path("name") substitutes into the {name} slot in the path. The base URL must end in /.
B3Domain models — the flat shape your UI wantsPokemon.kt

Plain data classes, no serialization annotations: a PokemonSummary(dexNumber, name) for the list and a PokemonDetail(dexNumber, name, heightMeters, weightKg, types) for the screen. The UI and ViewModel depend only on these, never on the DTOs — so if PokéAPI restructured its JSON, only B1 and B4 would change.

B4Mapper — decode the server's valuesPokemon.kt

The single place wire-shape becomes app-shape — and the only place the API's quirks get translated:

  • Dex number from the url — parse the trailing integer out of .../pokemon/25/ for each list summary (or off the detail's id).
  • Units — height / 10.0 → metres; weight / 10.0 → kilograms.
  • Types → emoji — map each type name to a glyph for the UI (the course's no-icon-library convention; see the reference card). e.g. grass→🌿, fire→🔥, water→💧, electric→⚡, poison→☠️.

Think about: why is parsing the id from the url mapper work and not UI work? Which of these three conversions are you about to write a unit test for in Part G?

Part C. The offline cache — Room Android Studio

Same Room you built in RoomAndPreferences / StorageJourney: an @Entity, a @Dao, and a singleton @Database. The cache is what makes the app work offline and what makes you a good PokéAPI citizen (it asks apps to cache rather than re-fetch).

Persist what you fetch so it survives going offline
C1Entity + DAOdata/CachedPokemon.kt · PokemonDao.kt

An @Entity table keyed by Pokémon name (the natural primary key). Store the summary fields (dex number, name) always; store the detail fields (height, weight, types) too, nullable — they stay null until that Pokémon's detail has been fetched. The DAO needs: a reactive observeAll(): Flow<List<CachedPokemon>> ordered by dex number, a suspend read of one row by name, and @Insert(onConflict = REPLACE) upserts for one row and for a whole list.

Why a Flow read? Exactly as in RoomAndPreferences: a Flow query is live — when you upsert fresh data, Room re-emits and the list redraws itself with zero manual refresh. Store the type list as a single String (e.g. comma-joined) to avoid a @TypeConverter for this first cut.
C2Database singletondata/PokemonDatabase.kt

An abstract class PokemonDatabase : RoomDatabase() annotated @Database(entities = [CachedPokemon::class], version = 1), exposing abstract fun pokemonDao(), with the same double-checked-locking getInstance(context) singleton as NoteDatabase.

Part D. The repository — offline-first, real & fake Android Studio

One source of truth that hides where data comes from. It reads the cache, refreshes from the network, writes results back, and degrades gracefully when offline. The real/fake split is the same one that lets NetworkParsing run with no internet.

Combine network + cache behind one interface
D1The contractPokemonRepository.kt

An interface PokemonRepository with: fun observePokemon(): Flow<List<PokemonSummary>> (the cached list the list screen reads), a suspend fun refreshList() (fetch endpoint 1, upsert all), and a suspend fun getDetail(name): PokemonDetail (return cached detail if present, else fetch endpoint 2, upsert, return it). suspend functions may throw — the ViewModel turns that into an Error state.

D2RealPokemonRepository — offline-firstPokemonRepository.kt

Holds the PokemonApi and the PokemonDao. observePokemon() just maps the DAO's Flow of entities to domain summaries — the UI always reads the cache, so it shows instantly and works offline. refreshList() calls the API and upserts; if it throws (no network), the already-cached Flow keeps feeding the UI. getDetail() is cache-then-network.

Think about: "offline-first" means the UI never waits on the network to show something. Which method reads the cache, and which one fills it? What should happen on the very first launch with the cache empty and no network — and which state does the user see then?

D3FakePokemonRepository — no network, for tests & offline demoPokemonRepository.kt

Returns a few hardcoded Pokémon after a small delay() (so you can see the spinner) — no Retrofit, no Room. It must implement the whole PokemonRepository contract — in particular observePokemon() must actually emit its hardcoded summaries (e.g. flowOf(list)), because the list ViewModel (E1) and its test (G5) reach Success by collecting that Flow. Provide providePokemonRepository(context, useFake = false) as the one-line switch, default real (the API is free + keyless, so the app just works live), useFake = true for fully-offline runs and unit tests. This mirrors provideWeatherRepository().

Part E. ViewModels & sealed UI state Android Studio

Two screens, two ViewModels, the same Loading/Success/Error pattern you wrote in WeatherViewModel and MvvmState. The sealed type makes the when in the UI exhaustive, so you can't forget to render a state.

Expose each screen's state as a StateFlow
E1PokemonListViewModelStateFlow + sealed UiStatePokemonListViewModel.kt

Define sealed interface ListUiState { Loading; data class Success(val pokemon: List<PokemonSummary>); data class Error(val message: String) }. In init, set Loading, then collect the repository's observePokemon() Flow into Success; also call refreshList() in a try/catch that publishes Error only when the cache is empty. Expose a private MutableStateFlow as a public read-only StateFlow and a retry().

Cache + network, one state: if the cached Flow has rows, show Success even when the refresh fails — the user sees data, not an error. Reserve Error for "nothing cached and the fetch failed."
E2PokemonDetailViewModelload-by-key + retryPokemonDetailViewModel.kt

Takes the tapped Pokémon's name (passed from navigation). loadDetail() sets Loading, calls repository.getDetail(name) in viewModelScope.launch inside try/catch, and publishes Success(detail) or Error(message). A retry() just calls it again. Identical control flow to WeatherViewModel.loadWeather().

Part F. The Compose UI + Navigation 3 Android Studio

Two screens joined by a Nav3 back stack — the NavListDetail pattern with a different domain. Each screen is a pure function of its uiState, with an explicit branch for each of the three states.

Render the states; wire the back stack
F1Navigation keys + NavDisplayMainActivity.kt

Two @Serializable keys implementing NavKey (as in NavListDetail): data object ListKey and data class DetailKey(val name: String). In setContent: val backStack = rememberNavBackStack(ListKey), then a NavDisplay whose entryProvider has one entry<ListKey> and one entry<DetailKey>. Tapping a list row does backStack.add(DetailKey(name)); Back pops automatically.

Passing the name: the tapped Pokémon's name rides in the key (DetailKey(name)) — that's the Nav3 replacement for an Intent extra. Read it back inside the entry<DetailKey> { key -> … } block and hand it to the detail ViewModel.
F2ListScreenLazyColumn · 3 statesui/ListScreen.kt

Collect the list ViewModel's state with collectAsStateWithLifecycle() and when over it: Loading → a centred spinner; Error → the message + a Retry button; Success → a LazyColumn of rows (use items(..., key = { it.name })) showing the dex number + name, each row clickable to open the detail. A Scaffold + TopAppBar titled "PokéDex" frames it.

F3DetailScreentype chips · 3 states · Backui/DetailScreen.kt

Same three-state when on the detail ViewModel. Success shows a card: #NNN name, a row of type chips (emoji + label), and height (m) / weight (kg). The top bar shows a Back arrow that pops the stack. Error shows the reason + Retry.

Icons = emoji. As in Homework 7, use emoji inside a Text for the type glyphs, not Icons.Filled.* — the material-icons artifact isn't reliably available in this course's setup, and emoji keep the build green.

Done when…

  • the app launches to a live list of 151 Pokémon fetched from PokéAPI;
  • tapping a row navigates to that Pokémon's detail (number, types as emoji, height, weight);
  • both screens show a spinner while loading and a Retry on error;
  • killing the network (airplane mode) and reopening still shows the cached list and any Pokémon you'd viewed;
  • the system Back button returns from detail to list, and no template/placeholder text remains.

Part G. Unit tests Android Studio

Prove the pure parts on the plain JVM — no emulator, no network. Test the mapper directly, and test the ViewModels against the fake repository with runTest (the same kotlinx-coroutines-test you added in Part A).

Verify the logic that doesn't need a device (JUnit 4)

At least six tests covering, at minimum:

  • Mapper · units — a DTO with height = 7 maps to 0.7 m; weight = 69 → 6.9 kg.
  • Mapper · dex number — the url .../pokemon/25/ parses to dex number 25.
  • Mapper · types → emoji — "grass" maps to your grass glyph; an unknown type falls back, not crash.
  • Decoding — feeding the trimmed JSON sample to Json { ignoreUnknownKeys = true } decodes without throwing.
  • List ViewModel — with the fake repo and runTest, state ends in Success with the expected Pokémon.
  • Detail ViewModel — a fake that throws drives the state to Error; a normal fake drives it to Success.
One extra step for the ViewModel tests: a ViewModel launches work in viewModelScope, which dispatches on Dispatchers.Main — and there is no Main dispatcher on a plain JVM test, so a naïve ViewModel test throws "Module with the Main dispatcher had failed to initialize." Swap in a test dispatcher around each test: Dispatchers.setMain(StandardTestDispatcher()) in an @Before and Dispatchers.resetMain() in @After (the small "MainDispatcherRule" pattern), using the kotlinx-coroutines-test you already added in Part A. The mapper and decoding tests (the first four) need none of this — they're pure.
Run them: right-click the test file ▸ Run, or ./gradlew testDebugUnitTest. All green is the deliverable.
Deliver: a passing JUnit run (panel screenshot or BUILD SUCCESSFUL).
