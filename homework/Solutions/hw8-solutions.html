One idea carries the whole capstone: offline-first The UI never waits on the network to show something. The list screen reads the Room cache (a live Flow); the network only refreshes that cache in the background. So the app shows instantly, survives going offline, and a successful refresh re-renders the screen for free. Every decision below — the Flow read, the IGNORE insert, cache-then-network detail — serves that one idea.

Part B. The network layer

DTOs mirror the JSON; domain models are the flat shape the UI wants; the mapper converts between them. (If you built from scratch, B1–B3 are yours too; the starter provides them so you can focus on the mapper, B4.)

B1–B2 — DTOs & the Retrofit APIAndroid Studio

Two suspend GETs against PokéAPI — a list and one detail — decoded by kotlinx.serialization. Identical in shape to NetworkParsing's WeatherApi.

Where: Pokemon.kt (DTOs) · PokemonApi.kt

// @Serializable DTOs mirror the JSON exactly (the nested type wrapper too):
@Serializable data class PokemonListResponseDto(val results: List<NamedResourceDto>)
@Serializable data class NamedResourceDto(val name: String, val url: String)
@Serializable data class PokemonDetailDto(
    val id: Int, val name: String, val height: Int, val weight: Int, val types: List<TypeSlotDto>)
@Serializable data class TypeSlotDto(val slot: Int, val type: TypeRefDto)
@Serializable data class TypeRefDto(val name: String)

interface PokemonApi {
    @GET("pokemon")
    suspend fun listPokemon(@Query("limit") limit: Int = 151, @Query("offset") offset: Int = 0): PokemonListResponseDto

    @GET("pokemon/{name}")
    suspend fun getPokemon(@Path("name") name: String): PokemonDetailDto
}

// ignoreUnknownKeys = true drops the dozens of fields we don't model -> no crash.
private val json = Json { ignoreUnknownKeys = true }
fun providePokemonApi(): PokemonApi =
    Retrofit.Builder()
        .baseUrl("https://pokeapi.co/api/v2/")          // MUST end in "/"
        .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
        .build().create(PokemonApi::class.java)

Why it works. @Query appends ?limit=151; @Path("name") fills the {name} slot. suspend makes Retrofit run the I/O off the main thread. ignoreUnknownKeys is the one setting that keeps decoding alive against a big real-world payload.

Common mistake: a baseUrl without the trailing / (Retrofit throws), or omitting ignoreUnknownKeys so the first unmodelled field blows up decoding.
B4 — The mapper (the parsing payoff)Android Studio

Turn the network's quirks — a number hidden in a URL, decimetres, hectograms, a nested type wrapper — into clean domain values. These functions are pure, which is what lets Part G test them instantly.

Where: Pokemon.kt

// "https://pokeapi.co/api/v2/pokemon/25/" -> 25
// trimEnd drops the trailing slash; substringAfterLast grabs the last segment;
// toIntOrNull ?: 0 means a malformed url can never crash the list.
fun dexNumberFromUrl(url: String): Int =
    url.trimEnd('/').substringAfterLast('/').toIntOrNull() ?: 0

// A type NAME -> an emoji (the course's no-icon-library convention). The `else`
// branch is the important part: an unknown/new type degrades, it does not crash.
fun typeEmoji(typeName: String): String = when (typeName) {
    "normal" -> "⚪"
    "fire" -> "🔥"
    "water" -> "💧"
    "grass" -> "🌿"
    "electric" -> "⚡"
    "poison" -> "☠️"
    // … 12 more types …
    else -> "❓"
}

// One list entry -> a summary (dex number parsed from the url).
fun NamedResourceDto.toSummary(): PokemonSummary =
    PokemonSummary(dexNumber = dexNumberFromUrl(url), name = name)

// The detail DTO -> the flat domain model, with the UNIT conversions.
fun PokemonDetailDto.toDetail(): PokemonDetail =
    PokemonDetail(
        dexNumber = id,
        name = name,
        types = types.map { PokemonType(it.type.name, typeEmoji(it.type.name)) },
        heightMeters = height / 10.0,   // wire value is DECIMETRES (7 -> 0.7 m)
        weightKg = weight / 10.0,       // wire value is HECTOGRAMS (69 -> 6.9 kg)
    )

Why it works. Parsing the id, decoding the units, and translating the type enum all happen in one place, so the UI/ViewModel never touch the wire format. Using / 10.0 (a Double), not / 10 (integer division, which would floor 7 to 0), is what makes 0.7 m come out right.

Common mistake: integer division (height / 10) — it truncates to 0. Or forgetting the else branch in typeEmoji, so a new PokéAPI type throws instead of showing a neutral glyph.

Part C. The Room cache

C1/C2 — Entity, DAO, databaseAndroid Studio

A table keyed by name; summary columns always present, detail columns nullable until fetched. The DAO needs a reactive list read, a by-name read, and inserts.

Where: data/CachedPokemon.kt · data/PokemonDao.kt

@Entity(tableName = "cached_pokemon")
data class CachedPokemon(
    @PrimaryKey val name: String,   // names are unique in PokéAPI -> the natural key
    val dexNumber: Int,             // always known from the list summary
    val heightMeters: Double? = null,  // detail-only; NULL until /pokemon/{name} is fetched
    val weightKg: Double? = null,
    val typesCsv: String? = null,      // e.g. "grass,poison" (CSV avoids a @TypeConverter for now)
)

@Dao
interface PokemonDao {
    @Query("SELECT * FROM cached_pokemon ORDER BY dexNumber ASC")
    fun observeAll(): Flow<List<CachedPokemon>>            // reactive: re-emits on every change

    @Query("SELECT * FROM cached_pokemon WHERE name = :name LIMIT 1")
    suspend fun getByName(name: String): CachedPokemon?

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun upsert(pokemon: CachedPokemon)               // full row (used by detail)

    // THE KEY CHOICE: IGNORE, not REPLACE. A list refresh only knows name+dexNumber,
    // so REPLACE would null out the height/weight/types of any already-detailed row.
    @Insert(onConflict = OnConflictStrategy.IGNORE)
    suspend fun insertNewSummaries(pokemon: List<CachedPokemon>)
}

Why it works. A Flow query is live — the moment the repository writes fresh rows, Room re-runs the SELECT and pushes a new list, so the UI redraws itself. The two different conflict strategies are deliberate: REPLACE writes a full hydrated row on a detail fetch; IGNORE lets a list refresh add new Pokémon without clobbering details you already cached.

Common mistake: using one REPLACE "upsertAll" for the list — re-fetching the list then wipes every cached detail back to null, and Pokémon you've already viewed lose their height/weight/types.

Part D. The repository — offline-first

D2 — RealPokemonRepositoryAndroid Studio

Read the cache for the list; refresh from the network; for a detail, return the cached one if it's hydrated, else fetch + cache. May throw on no network — the ViewModel turns that into an Error state.

Where: PokemonRepository.kt

class RealPokemonRepository(
    private val api: PokemonApi,
    private val dao: PokemonDao,
) : PokemonRepository {

    // D2a — the UI reads the CACHE. Map Room's live Flow of rows to domain summaries;
    // a successful refresh re-emits and the screen redraws for free.
    override fun observePokemon(): Flow<List<PokemonSummary>> =
        dao.observeAll().map { rows -> rows.map { PokemonSummary(it.dexNumber, it.name) } }

    // D2b — fetch the list, add only NEW rows (IGNORE), preserving cached details.
    // If this throws (offline), the already-cached Flow keeps feeding the UI.
    override suspend fun refreshList() {
        val response = api.listPokemon()
        val rows = response.results.map { dto ->
            val summary = dto.toSummary()
            CachedPokemon(name = summary.name, dexNumber = summary.dexNumber)
        }
        dao.insertNewSummaries(rows)
    }

    // D2c — cache-then-network: return the hydrated cache row if present, else fetch
    // it, write the FULL row (REPLACE), and return it.
    override suspend fun getDetail(name: String): PokemonDetail {
        dao.getByName(name)?.toDetailOrNull()?.let { return it }   // cache hit
        val detail = api.getPokemon(name).toDetail()               // miss -> network
        dao.upsert(detail.toCacheRow())                            // hydrate the cache
        return detail
    }
}

// The typesCsv round-trip. toDetailOrNull returns null for a summary-only row, so
// getDetail knows it must hit the network.
private fun CachedPokemon.toDetailOrNull(): PokemonDetail? {
    if (heightMeters == null || weightKg == null || typesCsv == null) return null
    val types = typesCsv.split(",").filter { it.isNotBlank() }
        .map { PokemonType(it, typeEmoji(it)) }
    return PokemonDetail(dexNumber, name, types, heightMeters, weightKg)
}
private fun PokemonDetail.toCacheRow() = CachedPokemon(
    name = name, dexNumber = dexNumber, heightMeters = heightMeters, weightKg = weightKg,
    typesCsv = types.joinToString(",") { it.name },
)

Why it works. The UI subscribes to observePokemon() — the cache — so it shows instantly and offline. refreshList() is the only thing that touches the network for the list, and its failure is harmless because the cached Flow keeps emitting. getDetail() avoids a network call entirely once a Pokémon is cached. The provider defaults to this real repository (useFake = false); the fake stays for offline demos and tests.

Common mistake: having the ViewModel call the API directly and only "cache" as an afterthought. Then offline = blank screen. Offline-first means the UI's source of truth is the cache, full stop.

Part E. ViewModels & sealed state

E1 — PokemonListViewModelAndroid Studio

Expose a sealed Loading/Success/Error as a StateFlow; collect the cache into Success and refresh in the background; only show Error when there's nothing cached.

Where: PokemonListViewModel.kt

sealed interface ListUiState {
    data object Loading : ListUiState
    data class Success(val pokemon: List<PokemonSummary>) : ListUiState
    data class Error(val message: String) : ListUiState
}

// PLAIN ViewModel that TAKES its repository — this is what makes it JVM-testable
// (a test injects a fake; the screen injects the real one via factory()).
class PokemonListViewModel(private val repository: PokemonRepository) : ViewModel() {
    private val _uiState = MutableStateFlow<ListUiState>(ListUiState.Loading)
    val uiState: StateFlow<ListUiState> = _uiState.asStateFlow()

    init {
        repository.observePokemon()                                  // the CACHE
            .onEach { list -> if (list.isNotEmpty()) _uiState.value = ListUiState.Success(list) }
            .launchIn(viewModelScope)
        refresh()                                                    // the NETWORK
    }

    fun refresh() = viewModelScope.launch {
        if (_uiState.value !is ListUiState.Success) _uiState.value = ListUiState.Loading
        try {
            repository.refreshList()
        } catch (e: Exception) {
            if (_uiState.value !is ListUiState.Success)              // cache empty AND fetch failed
                _uiState.value = ListUiState.Error(e.message ?: "Couldn't load Pokémon.")
        }
    }

    companion object {
        fun factory(repository: PokemonRepository): ViewModelProvider.Factory =
            object : ViewModelProvider.Factory {
                @Suppress("UNCHECKED_CAST")
                override fun <T : ViewModel> create(modelClass: Class<T>): T =
                    PokemonListViewModel(repository) as T
            }
    }
}

Why it works. The sealed interface makes the UI's when exhaustive. Collecting the cache into Success and only writing Error when there's nothing to show is the offline-first rule expressed in state. The PokemonDetailViewModel is the same shape with a load(name) instead of the init-collect. (The detail VM is identical in spirit — see the project.)

Common mistake: showing Error whenever refreshList() fails — even though the cache has data. The user would see an error over a perfectly good list. Guard on "!is Success".

Part F. The Compose UI + Navigation 3

F1/F2/F3 — Nav host, list, detailAndroid Studio

Two @Serializable NavKeys; a NavDisplay back stack; each screen a pure function of its state with a branch per Loading/Success/Error.

Where: MainActivity.kt · ui/ListScreen.kt · ui/DetailScreen.kt

// MainActivity.kt — the keys + the host (NavListDetail pattern)
@Serializable data object ListKey : NavKey
@Serializable data class DetailKey(val name: String) : NavKey   // the tapped name rides in the key

NavDisplay(
    backStack = backStack,                                       // rememberNavBackStack(ListKey)
    onBack = { backStack.removeLastOrNull() },
    entryProvider = entryProvider {
        entry<ListKey> { ListScreen(onOpen = { name -> backStack.add(DetailKey(name)) }) }
        entry<DetailKey> { key -> DetailScreen(name = key.name, onBack = { backStack.removeLastOrNull() }) }
    },
)

// ui/ListScreen.kt — build the real repository from a Context and INJECT it.
val context = LocalContext.current
val repository = remember(context) { providePokemonRepository(context) }
val viewModel: PokemonListViewModel = viewModel(factory = PokemonListViewModel.factory(repository))
val state by viewModel.uiState.collectAsStateWithLifecycle()

when (val s = state) {                                           // one branch per state
    is ListUiState.Loading -> /* centered CircularProgressIndicator */
    is ListUiState.Error   -> /* message + Retry button -> viewModel::refresh */
    is ListUiState.Success -> LazyColumn {
        items(s.pokemon, key = { it.name }) { p -> PokemonRow(p) { onOpen(p.name) } }
    }
}

Why it works. Navigation is just pushing a key (backStack.add(DetailKey(name))); the tapped name travels in the key — the Nav3 replacement for an Intent extra. The screen builds the repository once (remember(context)) and injects it through the VM factory, which is exactly what lets Part G test the VM with a fake. The detail screen calls LaunchedEffect(name) { vm.load(name) }.

Common mistake: calling viewModel.load(name) directly in the composable body instead of inside a LaunchedEffect(name) — it would re-fire on every recomposition.

Part G. Unit tests

G — Mapper, decoding & ViewModel testsAndroid Studio

Prove the pure mapper and the decoding on the plain JVM, then test the ViewModels against a fake repository.

Where: src/test/.../PokemonTest.kt

// Pure mapper tests — call the function, assert the output (delta for Doubles).
@Test fun `toDetail converts units decimetres to metres and hectograms to kg`() {
    val dto = PokemonDetailDto(1, "bulbasaur", height = 7, weight = 69,
        types = listOf(TypeSlotDto(1, TypeRefDto("grass")), TypeSlotDto(2, TypeRefDto("poison"))))
    val d = dto.toDetail()
    assertEquals(0.7, d.heightMeters, 0.001)
    assertEquals(6.9, d.weightKg, 0.001)
    assertEquals("🌿", d.types[0].emoji)
}

// ViewModel test — inject a fake; the dispatcher rule + runTest(rule.dispatcher)
// share one scheduler, so advanceUntilIdle() drives the VM's coroutines.
@get:Rule val mainDispatcherRule = MainDispatcherRule()

@Test fun `list ViewModel reaches Success`() = runTest(mainDispatcherRule.dispatcher) {
    val vm = PokemonListViewModel(FakePokemonRepository())
    advanceUntilIdle()
    assertTrue(vm.uiState.value is ListUiState.Success)
}

// The rule: swap Dispatchers.Main for a TestDispatcher (there is no Main on the JVM).
class MainDispatcherRule(val dispatcher: TestDispatcher = StandardTestDispatcher()) : TestWatcher() {
    override fun starting(d: Description) { Dispatchers.setMain(dispatcher) }
    override fun finished(d: Description) { Dispatchers.resetMain() }
}

Why it works. The mapper tests need nothing special — pure functions. The ViewModel tests do: a VM launches on viewModelScope (= Dispatchers.Main), which doesn't exist on a plain JVM. The MainDispatcherRule installs a TestDispatcher as Main; passing rule.dispatcher to runTest shares one scheduler so advanceUntilIdle() runs the VM's coroutines (and skips the fake's virtual delay). Injecting the repository is what makes the VM constructible in a test at all.

Common mistake: testing a VM with no Main-dispatcher rule — you get "Module with the Main dispatcher had failed to initialize." Or keeping the VM as an AndroidViewModel(app) that builds its own repository — then you can't construct it on the JVM without Robolectric, and you can't inject a fake.
